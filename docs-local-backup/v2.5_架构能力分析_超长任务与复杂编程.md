# 祖龙 v2.5 架构能力分析：超长任务与复杂编程支持评估

**日期**: 2026-04-17  
**背景**: 基于无限深度任务图谱 + 五阶段流水线架构 (v2.5) 的能力边界分析

---

## 一、超长任务的深度拆解与执行能力

### 1.1 结论

数据结构层面完全支持无限深度拆解，但当前有两个实际限制需要后续增强。

### 1.2 已具备的能力

1. **无限深度拆解** — 测试已验证 5 级嵌套（`o1_1_1_1_1`，depth=6），理论上无上限。LLM 拆出多少层，框架就写入多少层，执行逻辑不受影响。

2. **叶子节点精细执行** — 只有最底层的叶子被执行，拆得越细，每个叶子的推理范围越小、质量越高。这本质上就是"深度推理"——把大问题分解到足够小的粒度再逐个攻破。

3. **规划表全局感知** — 每个叶子执行时注入完整规划表，知道"整体任务做到哪了"、"前面完成了什么"、"后面还有什么"，不会跑偏。

4. **ReviewGate 审查** — 执行完成后 LLM 审查整体质量，发现遗漏可标记具体节点重做，最多重试 2 次。

### 1.3 当前限制

| 限制 | 说明 | 影响 | 可能的增强方向 |
|------|------|------|--------------|
| **一次性拆解** | Stage1 一次完成全部拆解，执行中不能追加新子节点 | 如果某个子任务执行时发现比预期复杂，无法动态再拆 | Stage3 中检测到复杂度高时触发"子拆解"，在叶子下方追加子节点 |
| **上下文窗口** | 规划表随节点数增长可能超出上下文窗口 | 超长任务（100+叶子）的规划表可能被截断 | 分级规划表（只展示当前大纲下的详细 + 其他大纲的摘要） |

### 1.4 对"深度推理"的理解

当前架构的"深度推理"是通过**粒度控制**实现的——把问题拆到足够小，每个叶子的推理就是在小范围内的高质量推理。这与 `DeepReasoningEngine` 的四步推理链（分析→假设→验证→结论）不同。两者可以组合：叶子执行时调用深度推理引擎做四步分析。

## 二、复杂项目编程支持能力

### 2.1 结论

当前架构提供了很强的基础框架，但要达到"实时掌控理解项目全局"的完整效果，还需要补充代码感知层。

### 2.2 当前架构已提供的机制

| 需求 | 架构中的对应机制 | 当前实现程度 |
|------|----------------|------------|
| 实时掌控理解项目全局 | `to_planning_table()` 规划表 + 前端图谱实时推送 | **已实现** — 每个子任务执行时都能看到完整任务结构和进度 |
| 轻松理解每个板块和文件的功能 | 大纲节点 (outline) 对应模块/板块，任务节点对应具体文件/功能 | **结构已有** — 但需要 LLM 在拆解时主动映射到代码文件 |
| 轻松理解系统架构 | 递归树的层级关系天然表达架构层次 | **结构已有** — 大纲=模块，任务=组件，子任务=具体实现 |
| 轻松定位每个文件位置 | TaskNode.desc 可存储文件路径信息 | **需增强** — 当前 desc 是自然语言描述，未与文件系统关联 |
| 局部注意力高质量推理 | 叶子节点 prompt 只包含当前任务 + 前置依赖结果 | **已实现** — 每个叶子在小范围内独立推理 |
| 全局注意力防止跑偏 | 规划表注入到每个子任务的 prompt | **已实现** — 规划表包含全部大纲和叶子的状态摘要 |

### 2.3 需要补充的代码感知层

要真正实现"复杂项目编程助手"，任务图谱需要与代码结构打通：

| 增强方向 | 具体做法 | 难度 |
|---------|---------|------|
| **代码-节点映射** | 大纲节点关联目录（`o1.meta.dir = "src/components/"`），叶子节点关联文件 | 中 |
| **文件注册表** | Stage3 执行后记录创建/修改的文件列表到 `node.result` 的结构化字段 | 低 |
| **增量再拆解** | Stage3 中叶子执行发现复杂度高时，自动触发 Stage1 对该叶子做子拆解 | 高 |
| **代码结构扫描** | Stage1 之前扫描项目目录结构，作为 DECOMPOSE_PROMPT 的额外上下文 | 中 |

### 2.4 核心优势：双注意力模型

当前架构对复杂编程任务的最大优势是**双注意力模型**：

```
全局注意力（规划表）: 防止子任务跑偏，确保整体一致性
  ↕ 互补
局部注意力（叶子 prompt）: 在小范围内高质量推理，避免上下文污染
```

这比传统的"把整个项目塞给模型"（上下文溢出）和"每次只给一个文件"（缺乏全局观）都要好。

**类比**：规划表给了模型"地图"，叶子 prompt 给了模型"放大镜"。

### 2.5 信息流示例

```
叶子 o1_1_2 的执行 prompt:

```

## 三、总体评估

### 3.1 当前架构的优势

1. **结构无限制** — 递归树深度无上限，LLM 决定拆解粒度
2. **执行与结构解耦** — 叶子节点执行模型让执行逻辑不受层级深度影响
3. **双注意力** — 全局规划表 + 局部叶子 prompt 的互补
4. **质量保障** — ReviewGate 审查 + 重试机制
5. **模型无关** — 导演模式 + 宽松解析，任何模型可接入
6. **可视化** — 前端实时图谱推送，用户可观察进度

### 3.2 需要增强的方向（按优先级）

| 优先级 | 方向 | 理由 |
|--------|------|------|
| P0 | 增量再拆解（执行中追加子节点） | 超长任务的核心需求 |
| P1 | 分级规划表（避免上下文溢出） | 100+叶子任务的可行性保障 |
| P1 | 代码-节点映射 | 编程场景的基础设施 |
| P2 | 文件注册表（执行后记录文件变更） | 项目全局理解的数据来源 |
| P2 | 代码结构扫描注入 | 拆解质量提升 |

### 3.3 核心结论

> 当前的无限深度递归树 + 双注意力模型为复杂任务提供了**正确的骨架**。
> 数据结构不是瓶颈（已无限深度），真正的挑战在于：
> 1. **动态性** — 能否在执行过程中根据需要追加子节点
> 2. **可扩展性** — 规划表能否在大规模任务下仍然有效
> 3. **代码感知** — 图谱能否与实际代码文件系统打通

这三个方向是将"任务图谱"从通用任务处理框架进化为"复杂项目编程助手"的关键路径。
