## 分析1: ExperienceRAG 被动化 — 任务循环与外界隔离问题

### 问题描述
> 所有事件路由到L1-B，L1-B先进行语义匹配等检索经验注入模型，复杂任务途中的规划执行等好像无法触发经验调用。概念模型进入任务循环是否意味着与外界（宏空间等）隔离了，能否破除这种隔离？

### 代码审查结果

#### 1.1 L1-B 是否在路由前注入经验？— **否**

L1-B 的 `scheduler_gatekeeper.py` 在路由消息前构建两种上下文：

- **`_build_local_context(text)`** (line 1579+): 提取关键词、意图分类、情感分析等**局部语义理解**
- **`_search_shared_context(text)`** (line 1700+): 搜索视觉/音频等**共享感知上下文**

**两者均不搜索 ExperienceRAG。** L1-B 将打包的 `local_context + shared_context_snapshot` 交给 L2，但包中不含经验数据。

#### 1.2 L2 推理引擎是否主动检索经验？— **否**

`inference_engine.py` 中的 `_retrieve_from_rag()` (line 2351-2395) **显式跳过 ExperienceRAG**：
```
只搜索: knowledge_rag + memory_rag
不搜索: experience_rag
```

#### 1.3 search_experience 工具是否可用？— **可用但被动**

| 检查项 | 结果 |
|--------|------|
| `SearchExperienceTool` 注册于 `ToolEngine` | 是 (tool_engine.py:515-524) |
| FC 循环 normal 模式工具列表包含它 | 是 (inference_engine.py:487-489) |
| FC 循环 planning 模式工具列表包含它 | 是 (通过 Modification C 注入) |
| system prompt 引导模型主动调用它 | **否** — 无任何提示 |
| _build_messages_with_history_async 注入经验 | **否** — 只注入 RAG/memory/navigation |

**结论**: `search_experience` 虽然在工具列表中，但模型没有被引导去使用它。在复杂规划任务中，模型不会自发地想到要"查查以前的经验"。

#### 1.4 任务循环是否与外界隔离？— **完全隔离**

FC 循环 `_generate_with_vllm_and_tools()` (line 468-1109) 是一个 **同步阻塞循环**：

```
while iteration < max_iterations:
    response = vllm_client.chat.completions.create(...)  # 阻塞调用
    if tool_call:
        result = execute_tool(...)  # 阻塞执行
        messages.append(result)     # 追加结果
    else:
        return response             # 退出循环
```

**隔离特征**:
- 循环期间**不检查 EventBus** — 新事件无法传入
- 循环期间**不检查 `_interrupt_flag`** — 中断信号被忽略 (flag 只在 line 3275 的 `_generate_with_real_model` 中检查，该方法不是 FC 循环)
- 循环期间**不刷新记忆/RAG/注意力** — messages 只追加工具结果，从不重建 system prompt
- **L1-B 的 `_queue_task_or_preempt`** 发布 SYSTEM_INTERRUPT 事件，但 FC 循环不监听

#### 1.5 能否破除隔离？

**理论路径** (按实施难度排序):

| 方案 | 描述 | 侵入性 |
|------|------|--------|
| A. System Prompt 引导 | 在 system_parts 中加入"遇到复杂决策时请先 search_experience"的指导 | 低 |
| B. 自动经验注入 | 在 `_build_messages_with_history_async` 中加回 ExperienceRAG 搜索 | 中 |
| C. FC 循环内定期刷新 | 每 N 轮工具调用后重建 system prompt (含经验/记忆/注意力) | 高 |
| D. 异步事件监听 | 在 FC 循环中加入 `await check_event_bus()` 检查点 | 高 |

## 分析2: 中间插入任务 — EventBus路由/KV Cache热交换/多任务并行

### 问题描述
> 检查EventBus路由架构、KV Cache热交换的支持设计、L2 BUSY状态打断机制、L2-backup任务委派、结果归一化机制

### 代码审查结果

#### 2.1 EventBus 事件路由 — **确认所有事件先到 L1-B**

`event_bus.py` (line 98-101) 的核心原则：
```python
# TSD v1.7 核心原则：所有事件路由到 L1-B 先处理
# 例外：L2_OUTPUT_STREAM 和 L2_THINKING_STEP 走低延迟直通
```

`scheduler_gatekeeper.py` (line 73-92) 订阅了所有关键事件类型：
- USER_INPUT → 主路由入口
- SYSTEM_INTERRUPT → 中断处理
- L2_OUTPUT → L2 输出转发
- 其他内部事件

**路由状态机** (line 338-366):
```
IDLE 状态 → 正常处理 → 转发给 L2
ACTIVE_TASK 状态 → 判断 WAITING/BUSY:
  - WAITING → 直接插入新任务
  - BUSY → 调用 _queue_task_or_preempt()
```

#### 2.2 _queue_task_or_preempt 中断流程 — **已实现但断链**

`scheduler_gatekeeper.py` line 1807-1840:
```python
def _queue_task_or_preempt(self, text):
    local_context = self._build_local_context(text)       # 1. 理解新任务
    shared_context = self._search_shared_context(text)     # 2. 搜索共享上下文
    packaged_task = {text, local_context, shared_context}  # 3. 打包
    event_bus.publish(SYSTEM_INTERRUPT, payload=packaged_task)  # 4. 发布中断
```

**问题**: 中断事件发布后的处理链断裂：

| 环节 | 实现状态 | 证据 |
|------|----------|------|
| L1-B 发布 SYSTEM_INTERRUPT | 已实现 | scheduler_gatekeeper.py:1840 |
| InferenceEngine 订阅 SYSTEM_INTERRUPT | 已实现 | inference_engine.py:146 `_on_interrupt` |
| _on_interrupt 设置 _interrupt_flag | 已实现 | inference_engine.py:3043 |
| FC循环检查 _interrupt_flag | **未实现** | FC循环(line 468-1109)中无任何 interrupt 检查 |
| _generate_with_real_model 检查 | 已实现但无关 | line 3275 — 这是本地模型生成方法，非FC循环 |

**致命断链**: `_interrupt_flag = True` 之后，FC 循环继续运行，完全无感知。

#### 2.3 KV Cache 热交换 — **完整设计脚手架，未真实集成**

`hotswap_scheduler.py` 实现了完整的架构设计：

| 组件 | 文件/行号 | 状态 |
|------|-----------|------|
| `TaskContext` 数据结构 | hotswap_scheduler.py:42-55 | 设计完成 |
| `HotSwapConfig` 配置 | hotswap_scheduler.py:58-67 | 设计完成 |
| `OutputGateway` 输出网关 | hotswap_scheduler.py:148-229 | 逻辑完整 |
| `L1B_HotSwapScheduler` 调度器 | hotswap_scheduler.py:232-684 | **逻辑完整但引擎是Mock** |
| `MiniL2Engine` | hotswap_scheduler.py:70-145 | **"模拟 L2 引擎 (用于测试逻辑)"** |
| `set_real_engine()` 接口 | hotswap_scheduler.py:295-301 | 已预留，未被调用 |

**关键发现**:
- `L1B_HotSwapScheduler` 的 `L2_PRIME` 和 `L2_BACKUP` 默认初始化为 `MiniL2Engine` (Mock)

- **无 vLLM API 集成**: 没有 `from vllm import ...` 或 KV Cache block table 操作

#### 2.4 热交换完整流程（设计层面）

```
新任务到来时 (on_user_input_stream):
  ↓
  L2_PRIME.is_busy? 
  ├─ 否 → 直接分配 KV Cache blocks → L2_PRIME 生成
  └─ 是 → L2_PRIME.interrupt() → 当前任务标记 INTERRUPTED
       ↓
  active_tasks 数量 >= switch_threshold?
  ├─ 否 → 正常执行新任务
  └─ 是 → _trigger_hot_swap():
         ├─ 所有活跃任务 → MIGRATED → 绑定 L2_BACKUP
         ├─ 移入 freeze_stack
         ├─ 清空 active_tasks
         └─ _start_backup_computation():
              └─ L2_BACKUP.generate() → intermediate_result
              └─ 状态设为 WAITING_OUTPUT
              
  OutputGateway 流式输出:
  ├─ 每个 token 经过 emit() 检查紧急信号
  ├─ STOP 信号 → 毫秒级切断
  └─ PAUSE → 缓冲输出，RESUME → 释放缓冲
  
  WAITING_OUTPUT 任务 → handover_and_output():
  └─ 通过 OutputGateway 逐字符输出 (归一化)
```

#### 2.5 L2-Backup 任务委派 — **设计完成，实际是 Fallback**

当前实际运行中的 backup 模型（`inference_engine.py:194-211`）是 **故障回退**:
```python
# CORE 模型失败时才使用 BACKUP
backup_client = OpenAI(api_key=..., base_url=LLM_BACKUP_API_URL)
# 仅在 _generate_with_backup() 中使用
```

**不是**并行计算引擎。

#### 2.6 结果归一化 — **设计完成，未真实运行**

`hotswap_scheduler.py` 的设计：
- L2_BACKUP 计算结果存入 `intermediate_result`，状态设为 `WAITING_OUTPUT`
- 不直接输出，等待 `handover_and_output()` 通过 `OutputGateway` 统一输出
- OutputGateway 支持紧急中断 (EmergencySignal.STOP)

**但**: 由于 `MiniL2Engine` 是 Mock，且 `set_real_engine()` 未被调用，这套归一化流程在生产环境中**不运行**。

#### 2.7 SnapshotManager / L2SnapshotInterface — **存在桥接层**

| 组件 | 位置 | 调用者 |
|------|------|--------|
| `SnapshotManager` | snapshot_manager.py | 单例，LRU 快照管理 |
| `L2SnapshotInterface` | l2_snapshot_interface.py | 桥接层 |
| `AttentionController` | attention_controller.py:258,294 | **实际调用者** |
| `InterruptHandler` | interrupt_handler.py:166 | 直接用 snapshot_manager |

`AttentionController` 通过 `L2SnapshotInterface` 可以:
- `freeze_current_task()` → 创建 ContextSnapshot
- `restore_context_snapshot()` → 恢复快照

但这条路径与 InferenceEngine 的 FC 循环**完全平行**，两者不互通。

#### 2.8 综合评估

```
设计完成度: ████████░░ 80%
实际集成度: ██░░░░░░░░ 20%
生产可用度: █░░░░░░░░░ 10%
```

## 分析3: 思维导航是否实现了上下文窗口机制

### 问题描述
> 思维导航是否实现了上下文窗口机制

### 代码审查结果

#### 3.1 AttentionWindowManager — **完整实现**

`pipeline/attention_window.py` (line 86-589) 实现了完整的上下文窗口管理：

**三种模式**:
| 模式 | 描述 | 效果 |
|------|------|------|
| `GLOBAL` | 全局视角 | 所有节点平等评分 |
| `FOCUS` | 聚焦模式 | 焦点节点及邻域获得高 mode_multiplier |
| `SINGLE_CHAIN` | 单链模式 | 只保留焦点链上的节点 |

**Token 预算机制** (line 127-130):
```python
budget = max(int((context_window_size - reserved) * 0.90), 1024)
```

**消息评分公式** (line 367-479):
```
score = base × time_decay × mode_multiplier × graph_boost
```
- base: 按角色类型 (system=1.0, user=0.9, tool_result=0.8, assistant=0.7)
- time_decay: 时间衰减 `max(0.3, 1.0 - age * decay_rate)`
- mode_multiplier: 模式加成 (FOCUS 模式焦点节点 1.5x)
- graph_boost: 图激活加成 (BFS 扩散激活)

**驱逐机制** (line 274-363):
- 按评分排序，低分消息被驱逐
- 被驱逐的消息生成摘要 (summary) 保留关键信息
- 保护 system 消息和最近 N 条消息

#### 3.2 谁使用了 AttentionWindowManager？

| 调用者 | 使用方式 | 文件 |
|--------|----------|------|
| `Orchestrator` | `apply_window()` 在每次 LLM turn 前调用 | pipeline/orchestrator.py:530 |
| `Orchestrator` | `observe_tool_call()` 在工具执行后调用 | pipeline/orchestrator.py:630 |
| **InferenceEngine FC循环** | **不使用** | inference_engine.py (无引用) |

#### 3.3 NavigateAttentionTool 与 AttentionWindowManager 的断裂

`attention_tool.py` 的 `NavigateAttentionTool` 实现了 deeper/broader/jump 导航：

```python
# NavigateAttentionTool 的操作:
def execute(direction, node_id):
    memory_graph.update_focus_to_node(node_id)  # 更新 MemoryGraph 焦点
    return focus_path_summary                    # 返回导航结果
```

**问题**: `navigate_attention` 工具只更新 `MemoryGraph._last_focus_context`，**不更新** `AttentionWindowManager._current_node_id`。

`AttentionWindowManager.observe_tool_call()` (line 202-235):
```python
def observe_tool_call(self, fn_name, fn_args, result):
    if 'node_id' in fn_args:
        self._current_node_id = fn_args['node_id']  # 从参数提取
```

但 NavigateAttentionTool 的 `fn_args` 格式是 `{"direction": "deeper", ...}`，并非总是包含 `node_id`，导致 `observe_tool_call` 无法同步焦点。

#### 3.4 两套焦点系统的断裂

```
系统 A: MemoryGraph._last_focus_context    ← NavigateAttentionTool 更新
         │
         └→ get_focus_path_summary()       ← _build_messages_with_history_async 注入 system prompt
         
系统 B: AttentionWindowManager._current_node_id  ← observe_tool_call 更新(不可靠)
         │
         └→ apply_window() 的评分和驱逐   ← Orchestrator 使用, FC循环不使用
```

**两个系统各自为政**: 导航更新了 A 但没同步 B；窗口管理使用 B 但 B 的焦点信息不准确。

#### 3.5 结论

| 能力 | 状态 |
|------|------|
| 上下文窗口 token 预算管理 | 已实现 (AttentionWindowManager) |
| 基于图激活的消息评分 | 已实现 (BFS 扩散 + graph_boost) |
| 消息驱逐与摘要 | 已实现 |
| Orchestrator 管线集成 | 已实现 |
| FC 循环（vLLM推理）集成 | **未实现** |
| NavigateAttentionTool 焦点同步 | **断裂** |
| 导航引起的窗口重新评分 | **不触发** |

## 分析4: 思维导航/动态记忆/工具注入的覆盖 vs 追加机制

### 问题描述
> 思维导航，动态记忆，工具注入等，是否会覆盖之前的注入内容，能否按需要保留之前的并且注入新需求的内容

### 代码审查结果

#### 4.1 _build_messages_with_history_async — **每次全量重建（覆盖模式）**

`inference_engine.py` line 2710-2905 的 `_build_messages_with_history_async()`:

```
每次被调用时:
  system_parts = []
  system_parts.append(identity_prompt)       # 身份
  system_parts.append(tool_instructions)     # 工具说明
  system_parts.append(manual_context)        # 操作手册
  system_parts.append(visual_context)        # 视觉上下文
  system_parts.append(rag_results)           # RAG 检索结果 ← 每次重新检索
  system_parts.append(search_results)        # 搜索结果
  system_parts.append(navigation_summary)    # 思维导航 ← 每次重新获取
  system_parts.append(memory_context)        # MemoryGraph 记忆 ← 每次重新检索
  system_parts.append(trailing_guidance)     # 尾部引导
  
  messages = [{"role": "system", "content": "\n".join(system_parts)}]
  messages.extend(conversation_history)
```

**关键**: 这是**完全重建**，不是增量追加。每次调用都从零构建 system prompt。

**但此方法只在 FC 循环入口被调用一次**，之后 FC 循环内部的 messages 是追加模式。

#### 4.2 FC 循环内部 — **追加模式**

FC 循环 `_generate_with_vllm_and_tools()` 内部:
```python
# 循环开始: messages 由 _build_messages 构建
while iteration < max_iterations:
    response = vllm_client.chat.completions.create(messages=messages)
    
    if tool_call:
        messages.append({"role": "assistant", "content": ..., "tool_calls": [...]})
        result = execute_tool(...)
        messages.append({"role": "tool", "content": result})  # 追加
        # 注意: 不重建 system prompt!
    
    # 规划模式修正消息也是追加:
    messages.append({"role": "user", "content": correction})  # 追加
```

**结果**: FC 循环期间:
- 工具结果: **追加** (assistant + tool 对不断累积)
- 规划修正: **追加** (最多 6 条修正消息，来自 3 次 nudge)
- System prompt: **不刷新** (始终是循环开始时的版本)
- 导航/记忆/RAG: **不刷新** (因为不重建 system prompt)

#### 4.3 工具 Schema 注入 — **消费即清除（无重复）**

```python
# Modification C (inference_engine.py:547-579):
if self._pending_tool_schemas:
    new_schemas = [s for s in self._pending_tool_schemas if s not in existing]
    tools.extend(new_schemas)
    self._pending_tool_schemas.clear()  # 消费后清除
```

使用集合去重 + 消费后清除，**不会重复注入**。

#### 4.4 上下文压缩 — **破坏性截断**

触发条件:
- CircuitBreaker 进入 RED 状态
- 消息总量达到安全硬上限

压缩策略:
```python
# 保留: system prompt + 最后 3 组 tool 交互
# 丢弃: 中间所有消息
# 后果: 前 N 轮的工具结果、规划讨论、修正历史全部丢失
```

#### 4.5 各注入类型的行为总结

| 注入来源 | 时机 | 模式 | 是否保留旧内容 |
|----------|------|------|----------------|
| 思维导航 (get_focus_path_summary) | 每次 _build_messages | 覆盖 | 否 — 每次重新获取当前焦点路径 |
| MemoryGraph 记忆 (retrieve_context) | 每次 _build_messages | 覆盖 | 否 — 按当前 query 重新检索 |
| RAG 检索结果 | 每次 _build_messages | 覆盖 | 否 — 按当前 query 重新检索 |
| FC 循环工具结果 | 每次工具调用后 | **追加** | **是** — 累积在 messages 中 |
| 规划修正消息 | 每次 nudge | **追加** | **是** — 最多 6 条 |
| 工具 Schema | 跨迭代注入 | 消费清除 | 否 — 单次使用 |
| 视觉/音频上下文 | 每次 _build_messages | 覆盖 | 否 |

#### 4.6 核心问题: FC 循环内不刷新 = 旧注入冻结

```
时间线:
  T0: _build_messages → system prompt 包含 [导航A, 记忆B, RAG-C]
  T1: 工具调用 navigate_attention → MemoryGraph 焦点变为 D
  T2: 工具调用 search_memory → 发现新记忆 E
  T3: 模型继续推理，但 system prompt 仍是 [导航A, 记忆B, RAG-C]
      ↑ 导航D 和记忆E 的结果只在 tool result 消息中，不在 system prompt 中
```

**后果**:
- 导航工具的结果（焦点路径摘要）只存在于工具返回消息中
- 模型必须从工具返回消息中自行理解导航变化
- 如果 20+ 轮后触发压缩，早期的导航/记忆工具结果被丢弃

#### 4.7 能否"保留旧内容并注入新内容"？

**当前不支持**。原因:

1. `_build_messages_with_history_async` 是覆盖模式 — 每次从零构建
2. FC 循环内不调用 `_build_messages` — 初始注入冻结不变
3. 如果改为循环内定期重建 — 旧的 system prompt 被新的覆盖
4. 如果想要"累积"式注入 — 需要新的机制:
   - 维护一个 `injected_context_history` 列表
   - 每次注入时 append 而非 replace
   - 配合 token 预算做定向驱逐

## 综合诊断

### 四个问题的共同根因

```
┌─────────────────────────────────────────────────┐
│                FC 循环是封闭黑箱                    │
│                                                   │
│  入口: _build_messages (一次性构建)                  │
│  循环: messages.append() (只进不出)                  │
│  出口: return final_response                       │
│                                                   │
│  不检查: EventBus / interrupt_flag                 │
│  不刷新: system prompt / 记忆 / 导航 / RAG          │
│  不同步: AttentionWindowManager / MemoryGraph       │
│  不中断: 无论外部发生什么                             │
└─────────────────────────────────────────────────┘
```

### 系统现状评估

| 模块 | 独立测试 | 生产集成 | 差距 |

### 优先修复建议
