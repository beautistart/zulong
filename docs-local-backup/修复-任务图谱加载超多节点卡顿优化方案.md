# 任务图谱加载超多节点时卡顿 - 性能优化方案

> 问题描述：Web页面加载超多节点任务图谱时出现明显卡顿
> 分析日期：2026年5月8日

---

## 一、问题诊断

### 1.1 当前架构分析

祖龙系统的任务图谱渲染已经具备了一定的性能优化机制：

```
当前渲染架构
├── 渲染模式自动切换（根据节点数量）
│   ├── ≤200节点: full（全量渲染）
│   ├── 200-500节点: full_optimized（优化渲染）
│   ├── 500-2000节点: virtual（虚拟化渲染）
│   └── >2000节点: degraded（降级渲染）
│
├── 布局计算（Web Worker）
│   ├── ≤200节点: hierarchy算法（层次布局）
│   └── >200节点: force算法（力导向布局）
│   └── 超时时间: 10秒
│
── 视口管理
│   ├── 空间索引（SpatialIndex）
│   ├── 可见性裁剪（只渲染视口内节点）
│   └── 节流机制（16ms）
│
└── Canvas渲染
    ├── requestAnimationFrame循环
    ├── 边渲染（全部）
    └── 节点渲染（可见区域）
```

### 1.2 性能瓶颈分析

通过代码审查，发现以下潜在瓶颈：

#### 瓶颈1：布局计算耗时过长

**位置**：`GraphCanvas.tsx` 第43-81行

**问题**：
- >200节点使用force算法（力导向布局），计算复杂度高 O(n²)
- 10秒超时可能不够（特别是5000+节点）
- 布局计算在主线程等待结果，阻塞UI

**影响**：
- 用户看到"正在计算布局..."长时间不动
- 可能触发超时错误

#### 瓶颈2：边渲染未优化

**位置**：`GraphCanvas.tsx` 第124行

```typescript
renderer.renderEdges(edgeArray, positions)  // ❌ 渲染所有边
```

**问题**：
- 即使启用虚拟化，仍然渲染**所有边**
- 如果节点数=2000，边数可能=4000-8000
- 边的渲染复杂度 O(edges)，没有视口裁剪

**影响**：
- 边成为主要性能瓶颈
- 降级模式也无法缓解

#### 瓶颈3：频繁的状态更新

**位置**：`GraphCanvas.tsx` 第83-94行

```typescript
useEffect(() => {
    const newMode = selectRenderMode(nodeCount)
    if (newMode !== renderMode) {
        setRenderMode(newMode)  //  每次节点数变化都检查
    }
}, [nodeCount, renderMode, setRenderMode])

useEffect(() => {
    if (nodeCount > 0 && renderStatus === "idle") {
        performLayout()  // ❌ 可能频繁触发
    }
}, [nodeCount, renderStatus, performLayout])
```

**问题**：
- 节点逐个添加时，每次都会触发重新布局
- 没有批量更新机制
- 没有防抖/节流

#### 瓶颈4：空间索引构建时机

**位置**：`GraphCanvas.tsx` 第70行

```typescript
viewportManagerRef.current.buildIndex(nodes)  // 布局完成后才构建
```

**问题**：
- 空间索引在布局完成后才构建
- 如果布局失败，没有降级索引
- 大规模节点时索引构建本身也耗时

#### 瓶颈5：Canvas渲染循环未优化

**位置**：`GraphCanvas.tsx` 第139-148行

```typescript
useEffect(() => {
    if (renderStatus === "rendering" || renderStatus === "completed" || renderStatus === "layouting") {
        animFrameRef.current = requestAnimationFrame(renderCanvas)
    }
    return () => {
        if (animFrameRef.current) {
            cancelAnimationFrame(animFrameRef.current)
        }
    }
}, [renderStatus, renderCanvas])
```

**问题**：
- `renderStatus` 变化时重新启动渲染循环
- `renderCanvas` 依赖项过多（nodes, edgeArray等）
- 可能频繁重建函数引用

## 二、优化方案

### 2.1 紧急优化（立即可实施）

#### 优化1：边渲染视口裁剪 ⭐⭐⭐⭐⭐

**优先级**：最高
**影响**：立竿见影，减少50-80%渲染开销

**修改文件**：`GraphCanvas.tsx`

```typescript
// 修改前（第124行）
renderer.renderEdges(edgeArray, positions)

// 修改后
const visibleIds = new Set(
  renderMode === "virtual" || renderMode === "degraded"
    ? viewportManagerRef.current.getVisibleNodeIds(currentViewport)
    : nodeArray.map(n => n.id)
)

// 只渲染可见节点相关的边
const visibleEdges = edgeArray.filter(e => 
  visibleIds.has(e.source) && visibleIds.has(e.target)
)

renderer.renderEdges(visibleEdges, positions)
```

**预期效果**：
- 2000节点场景：边渲染从8000条 → 500-1000条
- 帧率提升：15 FPS → 45+ FPS

#### 优化2：布局计算渐进式展示 ⭐⭐⭐⭐⭐

**优先级**：最高
**影响**：改善用户体验，避免长时间等待

**修改文件**：`GraphCanvas.tsx`

```typescript
// 新增：渐进式布局回调
const [partialLayout, setPartialLayout] = useState<Map<string, Position>>(new Map())

useEffect(() => {
    const client = layoutClientRef.current
    
    // 监听部分布局结果
    client.onPartial((positions) => {
        const posMap = new Map(Object.entries(positions))
        setPartialLayout(posMap)
        setLayoutPositions(posMap)
        setRenderStatus("rendering")  // 提前进入渲染状态
    })
    
    // 布局完成后合并
    client.onRefined((positions) => {
        const posMap = new Map(Object.entries(positions))
        setPartialLayout(new Map())  // 清除临时状态
        setLayoutPositions(posMap)
    })
    
    return () => {
        // 清理回调
    }
}, [])

const performLayout = useCallback(async () => {
    if (nodeCount === 0) return
    
    const client = layoutClientRef.current
    const algorithm = nodeCount <= 200 ? "hierarchy" : "force"
    
    try {
        setRenderStatus("layouting")
        const layoutNodes = nodeArray.map((n) => ({ id: n.id, parentId: n.parentId }))
        const layoutEdges = edgeArray.map((e) => ({
            source: e.source,
            target: e.target,
            type: e.type,
        }))
        
        // 增加超时时间（根据节点数动态调整）
        const timeoutMs = nodeCount <= 500 ? 10000 : 
                         nodeCount <= 2000 ? 20000 : 30000
        
        const positions = await client.requestLayout(layoutNodes, layoutEdges, {
            algorithm,
            width: width * 2,
            height: height * 2,
        }, timeoutMs)
        
        const posMap = new Map<string, Position>()
        for (const [id, pos] of Object.entries(positions)) {
            posMap.set(id, pos)
        }
        setLayoutPositions(posMap)
        
        viewportManagerRef.current.buildIndex(nodes)
        
        const fitViewport = viewportManagerRef.current.fitAll(nodes, width, height)
        setViewport(fitViewport)
        
        setRenderStatus("rendering")
    } catch (err) {
        console.error("Layout failed:", err)
        setErrorMessage(`布局计算失败: ${err instanceof Error ? err.message : String(err)}`)
        setRenderStatus("error")
    }
}, [nodeCount, nodeArray, edgeArray, nodes, width, height, setRenderStatus, setLayoutPositions, setViewport, setErrorMessage])
```

**预期效果**：
- 用户可以在1-2秒内看到初步布局（即使不完整）
- 布局逐渐优化，不会完全阻塞

#### 优化3：节点批量更新防抖 ⭐⭐⭐⭐

**优先级**：高
**影响**：减少不必要的重复布局

**修改文件**：`GraphCanvas.tsx`

```typescript
import { useRef, useCallback, useEffect } from "react"

const GraphCanvas = ({ width = 800, height = 600 }: GraphCanvasProps) => {
    // ... 其他代码
    
    // 新增：批量更新定时器
    const layoutTimerRef = useRef<ReturnType<typeof setTimeout> | null>(null)
    const pendingLayoutRef = useRef(false)
    
    const scheduleLayout = useCallback(() => {
        if (pendingLayoutRef.current) return
        
        // 清除之前的定时器
        if (layoutTimerRef.current) {
            clearTimeout(layoutTimerRef.current)
        }
        
        pendingLayoutRef.current = true
        
        // 防抖500ms
        layoutTimerRef.current = setTimeout(() => {
            pendingLayoutRef.current = false
            if (nodeCount > 0 && renderStatus === "idle") {
                performLayout()
            }
        }, 500)
    }, [nodeCount, renderStatus, performLayout])
    
    // 节点数量变化时调度布局
    useEffect(() => {
        scheduleLayout()
        return () => {
            if (layoutTimerRef.current) {
                clearTimeout(layoutTimerRef.current)
            }
        }
    }, [nodeCount, scheduleLayout])
    
    // ... 其他代码
}
```

**预期效果**：
- 节点逐个添加时，不会每次都触发布局
- 批量添加完成后统一布局一次
- 减少布局计算次数 70-90%

#### 优化5：增量布局 ⭐⭐⭐⭐

**问题**：
- 每次添加新节点都重新布局所有节点
- 已有节点的位置会大幅跳动

**优化方案**：

```typescript
// layout-worker.ts

function incrementalLayout(existingPositions, newNodes, edges, config) {
    // 只对新节点和受影响的节点进行布局
    // 已有节点位置保持稳定
    
    const affectedNodes = findAffectedNodes(newNodes, edges)
    const nodesToLayout = [...newNodes, ...affectedNodes]
    
    // 固定已有节点位置
    const fixedNodes = new Set(Object.keys(existingPositions))
    
    return forceDirectedLayout(nodesToLayout, edges, {
        initialPositions: {
            ...existingPositions,
            // 新节点从父节点附近开始
            ...initializeNewNodes(newNodes, existingPositions)
        },
        fixedNodes,  // 不移动这些节点
        maxIterations: 30
    })
}
```

**预期效果**：
- 添加10个节点：全量布局5秒 → 增量布局0.5秒
- 用户体验：节点平滑出现，不会跳动

#### 优化6：Canvas渲染性能优化 ⭐⭐⭐

**修改文件**：`SkeletonRenderer.ts`

**优化点**：

```typescript
class SkeletonRenderer {
    // 优化1：批量绘制（减少context切换）
    renderNodes(nodes: NodeAttribute[], selectedId: string | null): void {
        const ctx = this._ctx
        
        // 按状态分组，一次性绘制所有相同状态的节点
        const nodesByStatus = new Map<string, NodeAttribute[]>()
        for (const node of nodes) {
            if (!nodesByStatus.has(node.status)) {
                nodesByStatus.set(node.status, [])
            }
            nodesByStatus.get(node.status)!.push(node)
        }
        
        for (const [status, statusNodes] of nodesByStatus) {
            ctx.fillStyle = NODE_STATUS_COLORS[status]
            ctx.beginPath()
            
            for (const node of statusNodes) {
                const sx = node.position.x * this._viewport.zoom + this._viewport.offsetX
                const sy = node.position.y * this._viewport.zoom + this._viewport.offsetY
                
                const radius = Math.max(NODE_RADIUS * this._viewport.zoom, 4)
                ctx.moveTo(sx + radius, sy)
                ctx.arc(sx, sy, radius, 0, Math.PI * 2)
            }
            
            ctx.fill()  // 一次性绘制所有相同颜色的节点
        }
        
        // 单独绘制选中节点（高亮）
        if (selectedId) {
            // ...
        }
    }
    
    // 优化2：边的简化绘制
    renderEdges(edges: EdgeAttribute[], positions: Map<string, Position>): void {
        const ctx = this._ctx
        const zoom = this._viewport.zoom
        
        // 小缩放时使用直线（而非曲线）
        const useCurve = zoom > 0.3
        
        ctx.strokeStyle = "#555"
        ctx.lineWidth = Math.max(0.5, zoom * 0.5)
        ctx.beginPath()
        
        for (const edge of edges) {
            const source = positions.get(edge.source)
            const target = positions.get(edge.target)
            if (!source || !target) continue
            
            const sx = source.x * zoom + this._viewport.offsetX
            const sy = source.y * zoom + this._viewport.offsetY
            const tx = target.x * zoom + this._viewport.offsetX
            const ty = target.y * zoom + this._viewport.offsetY
            
            if (useCurve) {
                // 贝塞尔曲线（美观但慢）
                const cpx = (sx + tx) / 2
                const cpy = (sy + ty) / 2 - 20
                ctx.moveTo(sx, sy)
                ctx.quadraticCurveTo(cpx, cpy, tx, ty)
            } else {
                // 直线（快速）
                ctx.moveTo(sx, sy)
                ctx.lineTo(tx, ty)
            }
        }
        
        ctx.stroke()
    }
}
```

**预期效果**：
- 渲染速度提升 30-50%
- 减少context状态切换开销

#### 优化8：服务端布局预计算 ⭐⭐⭐

@router.post("/graph/layout")
async def compute_graph_layout(request: LayoutRequest):

    algorithm = request.algorithm
