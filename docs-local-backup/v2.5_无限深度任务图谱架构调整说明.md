# 祖龙 v2.5 架构调整说明：无限深度任务图谱与五阶段流水线

**日期**: 2026-04-17  
**版本**: v2.4.1 → v2.5  
**变更级别**: 重大架构升级

---

## 1. 变更概述

本次架构调整将祖龙系统的复杂任务处理能力从**扁平子任务列表**升级为**无限深度递归树 + 五阶段流水线**，这是自 v2.0 以来最大的架构变更。

### 1.1 核心变更

| 维度 | 变更前 (v2.4.1) | 变更后 (v2.5) |
|------|----------------|--------------|
| 架构位置 | `zulong/skill_packs/packs/complex_task/` (技能包) | `zulong/pipeline/` (框架骨架) |
| 任务结构 | 固定两层：大纲 → 子任务 | 无限深度：大纲 → 任务 → 子任务 → 子子任务 → ... |
| 执行单元 | 所有 `type="task"` 的节点 | 仅叶子节点（无子节点的非模板节点） |
| 审查机制 | 无 | ReviewGate + 最多 2 次重试循环 |
| 进度可视化 | 无 | 前端实时图谱推送 + 递归规划表 |
| 注意力模型 | 全局上下文平铺 | 局部注意力 (叶子 prompt) + 全局注意力 (规划表) |

### 1.2 为什么不是技能包

旧架构将 `ComplexTaskPack` 作为可装卸的技能包，带来以下问题：
- 核心流程插件化导致耦合不清
- 生命周期管理复杂（安装/卸载/内化）
- 无法与框架骨架深度集成（如前端图谱推送）

新架构将流水线作为**框架骨架**集成到 `zulong/pipeline/`，与 InferenceEngine 同级，不可装卸。

## 3. 无限深度递归树架构

### 3.1 双层架构：模板 + 动态生成

```
固定模板节点（框架自动创建，不可删除）:
  req (depth=0, type=requirement) ─── 原始需求锚点
   └── analysis (depth=1, type=analysis) ─── 需求分析结果

动态生成节点（LLM 按任务需要自行决定数量和深度）:
        ├── o1 (depth=2, type=outline)
        │    ├── o1_1 (depth=3, type=task)
        │    │    ├── o1_1_1 (depth=4, type=subtask)
        │    │    └── o1_1_2 (depth=4, type=subtask)
        │    └── o1_2 (depth=3, type=task)
        └── o2 (depth=2, type=outline)
             └── o2_1 (depth=3, type=task)
                  └── o2_1_1 (depth=4, type=subtask)
                       └── o2_1_1_1 (depth=5, type=subtask)
                            └── ... (无限)
```

### 3.2 深度→类型映射

| 深度 | 类型 | 语义 | 前端颜色 |
|------|------|------|---------|
| 0 | requirement | 原始需求根节点 | #8b5cf6 |
| 1 | analysis | 需求分析 | #6366f1 |
| 2 | outline | 大纲/模块 | #3b82f6 |
| 3 | task | 任务/步骤 | #06b6d4 |
| 4+ | subtask | 子任务（无限嵌套） | #10b981 |

### 3.3 叶子节点执行模型

**核心规则**: "谁被执行" 不由 `type` 决定，而由**结构**决定。

- 叶子节点 = 不在任何 h_edge 的 source 侧 且 不是模板节点
- 只有叶子节点进入 Stage3 执行
- 容器节点（有子节点的节点）仅作为结构分组
- 执行粒度完全由 LLM 拆解深度决定

**优势**: 无论 LLM 拆出多少层级，执行逻辑无需任何修改。

### 3.4 ID 命名规范

```
depth=2: o1, o2, o3, ...
depth=3: o1_1, o1_2, o2_1, ...
depth=4: o1_1_1, o1_1_2, o2_1_1, ...
depth=5: o1_1_1_1, o1_1_1_2, ...
...
```

规则: 深度 2 用 `o{index}`，之后递归 `{parent_id}_{index}`。

### 3.5 状态聚合

非叶子节点的 status 在 `to_frontend_dict()` 导出时动态计算：

```
聚合规则:
  全部 completed     → "completed"
  有 in_progress     → "in_progress"
  有 completed + pending → "in_progress" (混合视为进行中)
  有 blocked         → "blocked"
  有 needs_adjust    → "needs_adjust"
  全部 pending       → "pending"
```

这确保前端展示的容器节点状态始终反映其子树的真实进度。

## 4. 五阶段流水线详解

### 4.1 总体流程

```
用户输入 → ComplexityClassifier → is_complex?
                                    │
                                    ├── No → 标准 FC 推理循环
                                    └── Yes ↓
                                
PipelineOrchestrator.run(goal)
  │
  ├── 初始化 TaskGraph (req + analysis 模板节点)
  │
  ├── Stage1: Decompose (两相位)
  │    ├── Phase A: LLM 分析需求 → 填充 analysis 节点
  │    └── Phase B: LLM 多级拆解 → 六级递归解析 → 写入 TaskGraph
  │
  ├── Stage2: Plan (纯算法)
  │    └── 提取叶子 → 构建依赖图 → Kahn 拓扑排序 → 并行组
  │
  ├── Stage3: Execute
  │    └── 按并行组逐组执行叶子节点 (含规划表全局上下文)
  │
  ├── [循环] Stage4: ReviewGate
  │    ├── PASS → Stage5
  │    └── FAIL → 标记 needs_adjust → 重新 Stage3→4 (最多2次)
  │
  └── Stage5: Finalize
       └── 汇总叶子结果 → LLM 生成最终回答
```

### 4.2 Stage1 六级递归解析

这是整个流水线最复杂的组件，需要处理任意 LLM 输出格式：

| 优先级 | 策略 | 匹配格式 | 示例 |
|--------|------|---------|------|
| A | 嵌套 JSON | `[{"name":"...", "children":[...]}]` | 结构化输出 |
| B | Markdown 标题 | `## 大纲` → `### 任务` → `####` | 层级标题 |
| C | 多级编号 | `1.` → `1.1` → `1.1.1` | 数字嵌套 |
| D | 段落分节 | `大纲1: ... 大纲2: ...` | 关键词段落 |
| E | 扁平降级 | 编号/无序列表 | 包裹为单 outline |
| F | 终极兜底 | 任意文本 | 整段作为单节点 |

每个策略都支持**递归子节点**提取（`children`/`subtasks`/`steps` 等键名）。

### 4.3 默认依赖策略

```
同父节点下的叶子 → 顺序依赖 (o1_1_1 → o1_1_2)
跨父节点 → 无依赖 (o1_* 与 o2_* 可并行)
```

如果 LLM 输出中有显式依赖描述（"任务2依赖任务1"），优先使用显式依赖。

### 4.4 ReviewGate 宽松审查

```
PASS 信号: "全部通过" / "没有问题" / "质量合格" / "all pass"
FAIL 信号: "需要修改" / "不一致" / "遗漏" / "未完成"

判定逻辑:
  只有 PASS → PASS
  只有 FAIL → FAIL + 提取失败节点
  PASS + FAIL 同时 → 尝试定位失败节点，无法定位则默认 PASS
  都没有 → 默认 PASS (宽松策略，避免阻塞)
```

## 5. 局部注意力 + 全局注意力模型

### 5.1 双注意力机制

每个叶子节点在 Stage3 执行时，其 prompt 包含两部分：

**局部注意力** (专注当前任务):
- 叶子节点自身的 `label` 和 `desc`
- 前置依赖节点的 `result`（已完成的前置任务结果）

**全局注意力** (了解整体进展):
- `to_planning_table()` 生成的完整规划表
- 包含所有大纲、所有叶子的状态和结果摘要

### 5.2 信息流

```
叶子 o1_1_2 的执行 prompt:

[系统提示]
你是一个专业的任务执行助手。

[全局上下文 - 规划表]
## 当前任务规划
### 前端开发
| o1_1_1 | 按钮组件 | 完成 | 按钮组件已完成... |
| o1_1_2 | 表单组件 | 进行中 | |        ← 当前任务
| o1_2   | 样式调整 | 待开始 | |

[局部上下文]
当前任务: 表单组件
描述: 开发 React 表单组件
前置任务结果: 按钮组件已完成，包含 hover 效果...

[用户指令]
请执行上述任务。
```

## 6. 与 InferenceEngine 集成

### 6.1 复杂度路由点

在 `_on_l2_command()` 方法中，标准 Function Calling 推理之前插入复杂度判断。

路由条件:
- `text` 非空
- 非唤醒命令
- 非紧急事件
- 非视觉注意力事件

满足以上全部条件后，调用 `classify_complexity(text)` 打分，`score >= 0.4` 进入流水线。

### 6.2 降级保护

流水线失败时自动降级到标准推理，用户无感知：

```python
try:
    result = await orchestrator.run(goal)
    if result["success"]:
        publish_output(result["final_response"])
    else:
        raise RuntimeError("Pipeline failed")
except Exception:
    # 静默降级到标准推理
    await standard_inference(goal)
```

### 6.3 事件复用

复用现有 `L2_THINKING_STEP` WebSocket 事件通道，通过 `event_type` 前缀区分：

| event_type | 含义 |
|-----------|------|
| `pipeline.start` | 流水线开始 |
| `pipeline.stage_start` | 某阶段开始 |
| `pipeline.stage_done` | 某阶段完成 |
| `pipeline.graph_update` | 图谱结构更新 |
| `pipeline.node_update` | 单节点状态更新 |
| `pipeline.done` | 流水线完成 |

## 8. 迁移说明

### 8.1 API 变更

| 旧 API | 新 API | 说明 |
|--------|--------|------|
| `get_subtask_nodes()` | `get_leaf_nodes()` | 执行单元从类型判断改为结构判断 |
| 无 | `get_children(parent_id)` | 获取直接子节点 |
| 无 | `get_node_depth(node_id)` | 获取节点深度 |
| 无 | `depth_to_type(depth)` | 深度→类型映射 |
| 无 | `_aggregate_status(node_id)` | 递归状态聚合 |

`get_subtask_nodes()` 保留为兼容接口，内部调用 `get_nodes_by_type("task")`。
所有流水线阶段（plan/review/finalize）已迁移到 `get_leaf_nodes()`。

### 8.2 前端兼容

前端 `to_frontend_dict()` 输出格式不变（`nodes`, `hEdges`, `dEdges`），新增字段向后兼容：
- 非叶子节点的 `status` 字段现在是动态聚合值
- 新增 `pipeline.*` 事件类型（现有事件不受影响）
