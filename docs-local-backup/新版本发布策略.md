# 祖龙新版本发布策略

> **决策**: 在旧仓库更新（不创建新仓库）
> **理由**: 旧仓库Star=0，无社区关注，直接更新即可

---

## 一、为什么不创建新仓库

### 旧仓库状态
- Star: 0
- Fork: 0
- Watch: 0
- Tag: 1个

**结论**: 旧仓库没有社区关注度，创建新仓库没有意义，反而会：
- 失去Git历史连续性
- 需要重新配置所有设置
- 之前的链接失效

### 对比案例

| 项目 | 做法选择 | 理由 |
|------|---------|------|
| Vue 2 → Vue 3 | 同仓库不同分支 | 保持历史，渐进迁移 |
| Create-React-App | 同仓库更新 | 重大版本但同仓库 |
| 祖龙旧版 → 新版 | **同仓库更新** | 无社区关注，直接更新 |

## 五、关于旧版本的处理

### 选项A: 保留旧版本历史（推荐）

```
优点：
├─ 可以看到项目演进
├─ 旧版本代码仍可访问（通过历史commit）
└─ 透明度高

做法：
├─ 直接推送新版本
└─ 在README中说明："本项目已全面重构"
```

### 选项B: 删除旧版本痕迹

```
优点：
├─ 历史更干净
└─ 避免旧版本问题影响新版本

缺点：
├─ 丢失演进历史
└─ Git历史不连续

做法：
├─ git checkout --orphan new-main
├─ 重新提交
└─ 强制推送
```

**建议**: 选项A，保留历史。因为：
- Star=0，不用担心用户困惑
- 历史记录对开发者有价值
- 透明度高，增加信任
