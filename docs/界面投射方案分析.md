# Web端IDE界面投射方案分析

## 方案定义

### 用户真实方案（方案C：界面投射）

```
Web端触发编程任务
  ↓
自动打开本地IDE
  ↓
IDE执行所有操作（文件读写/命令执行）
  ↓
IDE插件界面投射到Web端 ← 关键创新
  ├─ WebSocket实时同步界面状态
  ├─ Web端渲染React组件（与IDE共享）
  └─ Web端操作转发到IDE执行
  ↓
用户在Web端操作，实际在IDE执行
```

**核心创新**：
- IDE = 执行引擎（后端）
- Web = 显示控制界面（前端）
- 界面投射而非屏幕投射

#### 3. 状态同步机制

```typescript
// Web端状态管理
class UIStateSync {
  private ws: WebSocket;
  private state: UIState;

  constructor() {
    this.ws = new WebSocket('ws://localhost:8090/ide-ui');
    this.ws.onmessage = (msg) => this.handleSync(msg);
  }

  // 接收IDE状态推送
  handleSync(msg: MessageEvent) {
    const data = JSON.parse(msg.data);
    if (data.type === 'ui_state_sync') {
      this.state = data.payload;  // 更新本地状态
      this.render();              // 触发渲染
    }
  }

  // 发送操作到IDE
  async sendAction(action: string, data: any) {
    this.ws.send(JSON.stringify({
      type: 'ui_action',
      payload: { action, data }
    }));
  }
}
```

### 方案C-2：WebRTC窗口投射

**原理**：通过WebRTC投射IDE窗口画面

```
IDE端
  ├─ Puppeteer/X11捕获窗口
  ├─ WebRTC视频流编码
  └─ WebSocket信令

Web端
  ├─ WebRTC视频接收
  ├─ 远程桌面式显示
  └─ 鼠标/键盘事件转发
```

**优点**：
- ✅ 完全无需开发UI组件
- ✅ IDE所有功能自动可用

**缺点**：
- ❌ 画质/延迟问题
- ❌ 无法复制粘贴文本
- ❌ 移动端操作困难
- ❌ 带宽消耗大

**结论**：不推荐，组件级投射更优

// 协议定义
interface UIProtocol {
  // IDE → Web：状态推送
  'ui_state_sync': {
    active_tab: 'chat' | 'mcp' | 'history' | 'settings';
    messages: Message[];
    task_graph: TaskGraphSummary;
    tool_calls: ToolCall[];
    fc_state: {
      turn: number;
      phase: 'running' | 'waiting_tool' | 'done';
    };
    files: FileTree;  // 文件树
    terminal_output: string;  // 终端输出
  };

  // Web → IDE：操作请求
  'ui_action': {
    action: 
      | 'send_message'
      | 'cancel_task'
      | 'approve_tool'
      | 'switch_tab'
      | 'open_file'
      | 'execute_command';
    data: any;
  };

  // IDE → Web：操作结果
  'ui_action_result': {
    action: string;
    success: boolean;
    result?: any;
    error?: string;
  };
}
```

last_state_hash = {}

async def smart_broadcast(session: IDESession):
    current_state = extract_ui_state(session)
    current_hash = hash_state(current_state)
    
    # 只在有变化时推送
    if current_hash != last_state_hash.get(session.session_id):
        await broadcast_ui_state(session)
        last_state_hash[session.session_id] = current_hash
```

# IDE端：监听文件变化
from watchdog.observers import Observer

class FileSystemSyncer:
    def __init__(self, ws_server):
        self.ws_server = ws_server
        self.observer = Observer()
        
    def on_modified(self, event):
        # 文件变化时推送
        asyncio.run_coroutine_threadsafe(
            self.ws_server.broadcast({
                "type": "file_change",
                "payload": {
                    "path": event.src_path,
                    "action": "modified"
                }
            }),
            self.ws_server.loop
        )

## 与原方案对比总结

### 原方案（方案B：自动打开IDE）

**流程**：
```
Web触发 → 打开IDE窗口 → 在IDE操作 → 查看结果
```

**问题**：
- ❌ 需要切换窗口
- ❌ Web端无法看到实时进度
- ❌ 分离的用户体验

### 你的方案（方案C：界面投射）

**流程**：
```
Web触发 → 打开IDE（后台） → Web端显示界面 → Web端操作
```

**优势**：
- ✅ 无需切换窗口
- ✅ Web端实时看到进度
- ✅ 统一的用户体验
- ✅ 保留IDE完整功能
- ✅ 本地执行性能

**创新点**：
- 界面与执行分离
- Web作为控制台，IDE作为执行引擎
- 最佳用户体验 + 最佳功能性能
