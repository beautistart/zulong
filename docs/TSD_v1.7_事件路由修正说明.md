# TSD v1.7 事件路由逻辑修正说明

**修订日期**: 2026-03-21  
**修订人**: 系统架构师  
**影响范围**: 事件总线、L1-B 守门员、L2 推理引擎

---

## 📋 核心变更

### 变更前的逻辑（❌ 旧）
```
用户事件 → EventBus
  ├─ SILENT → L1-B (唤醒检查)
  ├─ BUSY/WAITING → L1-B (中断决策)
  └─ IDLE → L2 (直接处理)  ✅ 正确
```

### 变更后的逻辑（✅ 新 - TSD v1.7）
```
用户事件 → EventBus
  ├─ SILENT → L1-B (唤醒检查)
  ├─ BUSY/WAITING → L1-B (中断决策 + 上下文打包 + 任务冻结)
  └─ IDLE → L2 (直接处理，不经过 L1-B)  ✅ 保持直连
```

---

## 🎯 设计原则

### 1. 空闲状态直连 L2（降低延迟）
- **场景**: L2 处于 `IDLE` 状态
- **路由**: `User Event → L2`
- **理由**: 
  - 无需中断决策
  - 无需任务冻结
  - 减少 L1-B 处理开销，降低响应延迟

### 2. 忙碌状态经 L1-B（智能管理）
- **场景**: L2 处于 `BUSY` 或 `WAITING` 状态
- **路由**: `User Event → L1-B → L2`
- **理由**:
  - 需要中断决策（是否打断当前任务）
  - 需要上下文打包（历史 + 当前双快照）
  - 需要任务冻结（保存当前进度）
  - 需要状态对比（检测环境变化）

### 3. 安静模式经 L1-B（节能过滤）
- **场景**: 系统处于 `SILENT` 模式
- **路由**: `User Event → L1-B`
- **理由**:
  - L1-B 是唯一有权唤醒 L2 的模块
  - 过滤非唤醒词指令（节能）
  - 检测紧急指令（特权穿透）

## 📊 状态机流转表（已更新）

| 当前状态 | 触发事件 | 路由目标 | L1-B 动作 | 新状态 |
|---------|---------|---------|----------|--------|
| SILENT | 唤醒词 | L1-B | `load_model()` | ACTIVE (L2 加载中) |
| SILENT | 普通指令 | L1-B | 丢弃/记录 | SILENT |
| **ACTIVE (IDLE)** | **用户指令** | **L2 (直连)** | **无** | **ACTIVE (BUSY)** |
| ACTIVE (BUSY) | 普通指令 (<2s) | L1-B | 去抖动丢弃 | ACTIVE (BUSY) |
| ACTIVE (BUSY) | 紧急指令 | L1-B | 中断决策 → 冻结 → 注入 | ACTIVE (中断处理) |
| **ACTIVE (WAITING)** | **用户指令** | **L1-B** | **中断决策 → 冻结 → 注入** | **ACTIVE (BUSY)** |
| ACTIVE | "睡觉"指令 | L1-B | `unload_model()` | SILENT |

## 📝 关键注释

1. **WAITING 状态**: L2 完成任务分段生成后，等待用户反馈的中间状态
   - 此时有新指令 → 需要 L1-B 中断决策
   - 因为可能涉及任务切换、上下文更新

2. **BUSY 状态**: L2 正在推理生成中
   - 此时有新指令 → 需要 L1-B 判断是否打断
   - 冷却时间 2.0s（去抖动）

3. **IDLE 状态**: L2 空闲，无任务执行
   - 此时有新指令 → 直接给 L2，无需 L1-B 介入
   - 降低延迟，提高响应速度

---

## 🎯 总结

**核心思想**: 
- 空闲时**直连 L2**（快）
- 忙碌/等待时**经 L1-B**（智能）
- 安静时**经 L1-B**（节能）

这样既保证了简单场景下的低延迟，又保证了复杂场景下的智能决策能力。
