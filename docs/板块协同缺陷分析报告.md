# 祖龙系统板块协同缺陷分析报告

> 分析日期: 2026-04-20
> 场景: 复杂任务执行中需访问RAG + 跨任务文件 + 中途被插入"帮我查天气" + 恢复原任务

---

## 一、场景全流程追踪

### 第一阶段：复杂任务启动

```
用户: "帮我开发一个3D射击游戏"
  |
L1B Gatekeeper.on_user_text()
  |- effective_state = IDLE -> _handle_normal_command()
  |- 构建 packaged_task {text, local_context, shared_context}
  +- event_bus.publish(L2_COMMAND)
  |
InferenceEngine._on_l2_command()
  |- _retrieve_from_rag(user_input)        <- 检索 knowledge + memory RAG
  |- _build_messages_with_history_async()
  |   |- 注入【参考知识】(RAG上下文)
  |   |- 注入【思维导航】(焦点路径摘要)
  |   +- 注入【记忆上下文】(MemoryGraph.retrieve_context)
  +- _generate_with_vllm_and_tools(messages, tools)
      |
FC循环第1轮: LLM 判断复杂任务 -> 调用 start_task_plan
  |- 创建 TaskGraph ("用户需求" + "需求分析")
  |- 创建 PlanningSession -> self._planning_session
  |- CB.escalate_for_planning()
  +- 注入规划工具schemas到 _pending_tool_schemas
      |
FC循环第2轮: 规划工具可用
  |- 模型应调用 plan_add_node 添加任务节点
  |- [如果模型生成文本] -> 纠正循环注入提醒 -> continue
  +- 重复调用 plan_add_node 多次
      |
FC循环第N轮: 规划完成
  +- 模型调用 submit_final_answer -> 清理 _planning_session
```

### 第二阶段：任务执行中访问知识RAG

```
FC循环中某轮: 模型需要技术参考
  |- 方式A: 自动注入 (每轮 _build_messages 都调用 _retrieve_from_rag)
  +- 方式B: 模型主动调用 search_experience FC工具
```

### 第三阶段：访问另一个任务中的文件

```
模型在规划中发现需要另一个任务的产出文件
  |- 需要: 找到之前任务的workspace目录和文件
  +- 当前能力: 无通道
```

### 第四阶段：用户中途插入"帮我查天气"

```
用户: "帮我查询一下今天天气"
  |
L1B Gatekeeper.on_user_text()
  |- effective_state = ACTIVE_TASK
  |- l2_status = ???
  |
  |- 如果 BUSY -> _queue_task_or_preempt()
  |   +- 消息排队, 等任务结束才处理
  |
  +- 如果 WAITING -> _handle_interruption_during_wait()
      |- freeze_current() 冻结当前任务
      |- clear_task()

```

### 第五阶段：天气查询处理

```
L2 收到天气查询 L2_COMMAND
  |
InferenceEngine._on_l2_command()
  |- self._planning_session = ??? (状态竞争)
  |- _retrieve_from_rag("今天天气")
  |- _build_messages_with_history_async() <- 构建新 messages
  +- _generate_with_vllm_and_tools()

```

### 第六阶段：恢复原任务

```
天气查询完成后
  |- 用户说"继续" -> _handle_resume_task()
  |   |- 从内存栈恢复 TaskSnapshot
  |   +- 或从磁盘恢复 SuspendableTaskState
  |
resume_suspended_task() [inference_engine.py:1132]
  |- 恢复 messages
  |- 恢复 CB 状态
  +- 注入恢复提示
  |
  直接调用 LLM.chat.completions.create() <- 不经过FC循环!
```

## 二、已定位的协同缺陷

### D1: TaskGraph 不写入 MemoryGraph [致命]

**涉及板块**: 图任务 x 图记忆

**现象**: `start_task_plan` 创建的 TaskGraph 不写入 MemoryGraph。

**代码证据**:
- `StartTaskPlanTool.execute()` (planning_tools.py:129) 创建 TaskGraph 并挂到 `engine._planning_session`
- 没有调用 `MemoryGraph.add_node()` 创建对应的 TASK 类型节点
- `DialogueAdapter` 的 `add_round()` 只记录对话, 不感知规划

**后果**:
- 图记忆中没有这个任务的存在记录
- 思维深度索引无法导航到任务节点
- 动态注意力机制无法基于任务节点计算激活值
- 如果任务中断后恢复, `retrieve_context()` 搜不到之前的规划上下文

### D2: FC循环内不刷新RAG/记忆/导航 [严重]

**涉及板块**: RAG x FC循环 x 注意力

**现象**: FC循环内部每轮迭代不重新构建 messages, 所以 RAG/记忆/思维导航的自动注入只发生一次 (进入FC循环前)。

**代码证据**:
- `_build_messages_with_history_async()` 在 inference_engine.py:2710 被调用一次
- FC循环 `_generate_with_vllm_and_tools()` 在 inference_engine.py:485 接收 messages, 之后每轮只 append tool_call/tool_result
- RAG上下文、记忆上下文、思维导航全部是静态的

**后果**:
- 模型在规划第10个节点时看到的记忆上下文还是第1轮注入的
- 思维导航焦点路径在整个FC循环中固化

### D3: 规划模式无 search_experience/navigate_attention 工具 [严重]

**涉及板块**: 规划 x 经验检索 x 注意力导航

**现象**: 修改 C 只注入7个规划工具, `search_experience` 和 `navigate_attention` 不在其中。

**代码证据**:
- inference_engine.py:547-579 只注入 `get_planning_tool_schemas()` 返回的7个规划工具

### D4: 跨任务文件访问通道不存在 [严重]

**涉及板块**: 图任务 x 文件系统 x 记忆

**现象**: 没有任何机制让模型发现和访问其他任务工作目录中的文件。

**代码证据**:
- `exec_write_file` 的 `_resolve_workspace_path()` 限制在当前 workspace_dir
- TaskGraph FileRef 只存储相对路径
- TaskSuspensionManager 元数据中有 workspace_dir 但无工具调用它

### D5: 历史任务在图记忆中不可搜索 [严重]

**涉及板块**: 图记忆 x 搜索 x 标签

承接 D1: NodeType.TASK 枚举存在但无代码写入任务节点。

### D6: FC循环中L2始终BUSY, 无法被中断 [致命]

**涉及板块**: 中断恢复 x L1B调度

**现象**: FC循环是同步/异步阻塞循环, L2 一直 BUSY, 用户插入消息只能排队。

**代码证据**:
- FC循环 (inference_engine.py:485-1109) 无 state_manager.set_l2_status(WAITING) 调用
- _handle_interruption_during_wait 只在 WAITING 触发

### D7: 排队消息处理后不自动恢复 [中等]

**涉及板块**: 中断恢复 x 任务栈

需要用户手动说"继续"。

### D8: PlanningSession 不在挂起快照中 [致命]

**涉及板块**: 任务中断 x 规划

**代码证据**:
- SuspendableTaskState 不含 planning_mode
- resume_suspended_task() 恢复时不重建 PlanningSession

### D9: 单例 InferenceEngine 的状态竞争 [致命]

**涉及板块**: 中断恢复 x 规划 x 引擎

_planning_session 是实例级状态, 新请求会覆盖或干扰旧状态。

### D10: resume 绕过FC循环, 无法继续使用工具 [致命]

**涉及板块**: 中断恢复 x FC循环 x 规划

**代码证据**:
```python
# inference_engine.py:1163-1177
response = self.vllm_client.chat.completions.create(
    model=LLM_MODEL_ID,
    messages=messages,
    # 注意: 没有 tools= 参数!
)
```

### D11: 恢复时不恢复焦点/注意力/记忆温度 [严重]

**涉及板块**: 中断恢复 x 注意力 x 深度索引

恢复代码只有3步: 恢复 CB、恢复 messages、注入提示。

## 三、根因分析

### 三个图系统独立, 无桥接层

```
MemoryGraph (图记忆)           TaskGraph (图任务)        思维深度索引 (焦点路径)
+----------------+            +----------------+       +----------------+
| DIALOGUE       |            | req            |       | L1 [session]   |
| KNOWLEDGE      |  <- 无桥 -> | analysis       | <- 无桥 -> | L2 [round]    |
| EXPERIENCE     |            | task_1         |       | L3 [focus]     |
| TASK (空壳)    |            | task_2         |       |                |
+----------------+            +----------------+       +----------------+
```

### FC 循环是密封的黑箱

进入后无法重新检索RAG、更新记忆、更新导航、响应外部中断。

### 任务中断-恢复路径是半成品

冻结不保存 PlanningSession/注意力/焦点; 恢复不走 FC 循环。

---

## 四、修复方向 (按优先级)

### P0 致命级

1. TaskGraph -> MemoryGraph 桥接
2. PlanningSession 纳入挂起快照
3. resume 走 FC 循环而非裸 LLM
4. FC 循环暴露中断点

### P1 严重级

5. 规划模式扩展工具集 (含 search_experience, navigate_attention)
6. FC 循环内周期性刷新记忆
7. 自动恢复机制

### P2 中等级

8. 跨任务文件发现工具
9. 恢复时重建注意力/焦点
