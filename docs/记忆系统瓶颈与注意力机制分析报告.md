### 1.3 节点审查循环阻塞分析

```python
async def _idle_review_loop(self):
    while self._running:
        await asyncio.sleep(300)  # 5分钟检查间隔
        
        # 检查点1: 系统空闲检测
        is_idle = state_manager.is_system_idle(_IDLE_THRESHOLD)
        if not is_idle:
            continue
        
        # 检查点2: L2状态检测（避免与用户请求竞争）
        if state_manager._l2_status not in (L2Status.IDLE, L2Status.WAITING):
            continue
        
        # 🔒 锁竞争点: 获取待审查候选列表
        with self._candidates_lock:
            pending = list(self._pending_llm_candidates)
        
        # 分批提交（每批10个）
        while pending:
            # 🔒 锁竞争点: 取出一批
            with self._candidates_lock:
                batch = list(self._pending_llm_candidates[:_IDLE_BATCH_SIZE])
            
            # 构建审查请求数据（访问节点属性）
            candidate_memories = []
            for cid in batch:
                node = self._nodes.get(cid)  # ← 可能与主线程竞争
                candidate_memories.append({...})
            
            # 提交到L2-BACKUP（异步，不阻塞）
            await reviewer.review_importance_candidates(candidate_memories)
            
            # 🔒 锁竞争点: 移除已审查的
            with self._candidates_lock:
                self._pending_llm_candidates = [
                    c for c in self._pending_llm_candidates 
                    if c not in reviewed_ids
                ]
        
        # 审查完成后触发保存
        self.save()  # ← 同步阻塞
```

#### 锁竞争场景

```
主线程                          审查线程
  │                               │
  ├─ add_node()                   │
  │   └─ with _candidates_lock:   │
  │        _pending_llm_candidates.append()  ←┐ 锁竞争
  │                               │           │
  │                               ├─ _idle_review_loop()
  │                               │   └─ with _candidates_lock:
  │                               │        pending = list(...)
  │                               │           ↑
  │                               │        等待主线程释放锁
```

**问题**: `_candidates_lock` 在以下场景会竞争：
1. **主线程添加节点**: `add_node()` → `_pending_llm_candidates.append()`
2. **审查线程取候选**: `_idle_review_loop()` → `list(_pending_llm_candidates[:10])`
3. **审查线程移除已审查**: `_pending_llm_candidates = [...]`

**影响**: 当主线程高频添加节点时，审查线程会被阻塞，导致审查积压。

### 1.5 瓶颈优化建议

| 优化项 | 方案 | 预期收益 | 实现难度 |
|--------|------|---------|---------|
| **增量保存** | 只序列化变更的节点，使用WAL追加写入 | 减少80%序列化时间 | 🔴 高 |
| **异步保存** | 将save()放入后台线程，主线程继续 | 消除主线程阻塞 | 🟡 中 |

| **FC期间禁止审查** | 在FC循环期间暂停节点审查提交 | 避免审查竞争LLM资源 | 🟢 低 |

### 2.2 为什么大上下文窗口下触发频率低

#### 压力计算公式

**代码位置**: `zulong/l2/pressure_detector.py:41-80`

```python
def calculate_pressure(self) -> PressureMetrics:
    total_tokens = sum(env.tokens for env in self._awm.envelopes)
    current_budget = self._awm.budget
    
    # 压力值 = 当前token / 预算
    pressure_value = total_tokens / current_budget
    
    # 预算 = (上下文窗口 - 预留) × 90%
    # 对于 DeepSeek-V4: budget = (131072 - 7096) × 0.9 = 111578
```

#### 不同上下文窗口下的压力对比

| 模型 | 上下文窗口 | Budget | 假设消息Token数 | 压力值 | 是否触发切换 |
|------|-----------|--------|----------------|--------|-------------|
| GPT-3.5 | 4,096 | 2,800 | 2,000 | 0.71 | ✅ 接近阈值 |
| GPT-4 | 8,192 | 5,900 | 4,000 | 0.68 | ✅ 接近阈值 |
| Claude-3 | 100,000 | 83,300 | 20,000 | 0.24 | ❌ 远低于阈值 |
| **DeepSeek-V4** | **131,072** | **111,578** | 30,000 | **0.27** | ❌ 远低于阈值 |

**结论**: 
- 当上下文窗口 ≥ 100K tokens时，典型对话（20-50轮）的token消耗很难触及压力阈值
- **DeepSeek-V4的131K窗口几乎不需要注意力切换**，除非任务极其复杂（数百轮对话）

```python
def _compute_transition(self, tool_name: str, tool_args: Dict) -> Optional[AttentionMode]:
    # 1. adjust_attention_mode 直接指定目标模式
    if tool_name == "adjust_attention_mode":
        return _mode_map.get(tool_args.get("mode", ""))
    
    # 2. 强制回全局
    if tool_name in _GLOBAL_FORCE_TRIGGERS:
        return AttentionMode.GLOBAL
    
    # 3. task_mark_status 根据 status 参数决定
    if tool_name == "task_mark_status":
        status = tool_args.get("status", "")
        if status in ("completed", "skipped"):
            return AttentionMode.GLOBAL
        if status == "in_progress" and self.mode == AttentionMode.FOCUS:
            return AttentionMode.SINGLE_CHAIN
    
    # 4. GLOBAL → FOCUS
    if self.mode == AttentionMode.GLOBAL and tool_name in _FOCUS_TRIGGERS:
        return AttentionMode.FOCUS
    
    # 5. FOCUS → SINGLE_CHAIN
    if self.mode == AttentionMode.FOCUS and tool_name in _SINGLE_CHAIN_TRIGGERS:
        return AttentionMode.SINGLE_CHAIN
    
    return None  # 保持当前模式
```

### 2.5 大上下文窗口下的注意力机制定位

#### 原始设计意图

注意力机制最初为小窗口模型（如GPT-3.5的4K）设计，目的是：
1. **裁剪无关消息**：在窗口快满时淘汰低权重内容
2. **聚焦关键上下文**：根据任务阶段动态调整关注点
3. **防止溢出**：避免超过模型最大上下文长度

#### 大窗口模型的实际表现

对于DeepSeek-V4 (131K上下文):
- **容量充裕**: 单次任务很难用完131K tokens
- **切换低频**: 压力值长期处于低位，不会触发压力驱动的切换
- **工具驱动**: 模式切换主要由工具类型决定，而非压力

#### 建议

| 场景 | 建议 |
|------|------|
| 大窗口模型 (≥100K) | 保持当前工具驱动模式，降低压力阈值或禁用压力检测 |
| 小窗口模型 (<10K) | 启用压力驱动切换，设置阈值=0.7 |

## 三、FC循环期间禁止节点审查提交

### 3.1 问题场景

```
用户发起任务 → FC循环启动 → LLM推理
                          │
                          ├─ 工具执行 → memory_graph.add_node()
                          │              └─ _pending_llm_candidates.append()
                          │
                          └─ 同时期：
                              _idle_review_loop() 检测到候选
                              └─ 提交审查任务到L2-BACKUP
                                  └─ 与主对话竞争LLM资源 ❌
```

### 3.2 解决方案

**核心思路**: 在FC循环运行期间，禁止节点审查提交

#### 实现要点

1. **状态标记**: 在`state_manager`中增加`fc_loop_running`标志
2. **审查条件**: `_idle_review_loop()`在提交前检查标志
3. **标志管理**: `ide_fc_runner`在进入/退出循环时设置/清除标志

#### 修改位置

| 文件 | 函数 | 修改内容 |
|------|------|---------|
| `zulong/core/state_manager.py` | `StateManager` | 新增 `fc_loop_running: bool` 属性 |
| `zulong/memory/memory_graph.py:1700-1791` | `_idle_review_loop()` | 提交前检查 `state_manager.fc_loop_running` |
| `zulong/ide/ide_fc_runner.py:1765-1811` | `_run_loop()` | 进入时设置 `fc_loop_running=True`，退出时清除 |

### A. MemoryGraph.save() 完整代码

```python
def save(self) -> bool:
    """保存记忆图谱到磁盘（原子写入：temp + rename，防止崩溃导致数据损坏）"""
    with self._save_lock:  # 🔒 阻塞点1
        try:
            filepath = os.path.join(self.persist_path, "memory_graph.json")
            temp_filepath = filepath + ".tmp"
            
            data = {
                "version": self._SCHEMA_VERSION,
                "nodes": {nid: node.to_dict() for nid, node in self._nodes.items()},  # 🔒 阻塞点2
                "edges": [...],
                "embeddings": self._serialize_embeddings(),
                "meta": {...},
            }
            
            # 写入临时文件 + fsync
            with open(temp_filepath, 'w', encoding='utf-8') as f:
                json.dump(data, f, indent=2, ensure_ascii=False)
                f.flush()
                os.fsync(f.fileno())  # 🔒 阻塞点3
            
            # 原子替换
            os.replace(temp_filepath, filepath)  # 🔒 阻塞点4
            
            # FAISS保存
            self._summary_index.save(sidecar_path)  # 🔒 阻塞点5
            
            return True
        except Exception as e:
            logger.error(f"[MemoryGraph] 保存失败: {e}")
            return False
```
