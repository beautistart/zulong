# 记忆架构改造任务文档

## 一、背景

祖龙系统引入了 MemoryGraph（记忆图谱）作为新一代统一记忆架构，但当前存在两个核心问题：

1. **旧记忆系统仍然独占 LLM 上下文注入通道**：`_build_messages_with_history_async()` 完全基于旧系统（conversation_history / ShortTermMemory / EpisodicMemory / RAG），MemoryGraph 只在推理结束后执行 `finalize_round()` 写入，从未被读取/查询——处于"只写不读"状态。
2. **MemoryGraph 仅做短期记忆**：所有节点统一 24h 半衰期衰减，无长期记忆固化能力，重要冷数据面临被误剪风险。

本文档定义将 MemoryGraph 改造为**完整记忆生命周期管理系统**的任务清单。

---

## 二、改造目标

- MemoryGraph 成为唯一的记忆中枢（读写统一）
- 通过**多维标签体系**（时间段/温度/重要度）替代硬分区，实现灵活的记忆分类与检索路由
- 废弃冗余旧组件，保留有价值的旧组件并打通
- 保证大规模记忆下的检索性能（热数据遍历 + 非热数据向量检索，并行执行）
- 复用旧系统已有的 LLM 审查、记忆强度、记忆巩固等成熟机制

---

## 三、旧组件处置清单

| 组件 | 处置 | 说明 |
|------|------|------|
| `MemoryRAG` | **废弃** | 与 DialogueAdapter + EpisodeAdapter 完全重叠，且有独立衰减机制与 MemoryGraph 冲突 |
| `ShortTermMemory` (SharedMemoryPool) | **废弃** | 对话存储/向量缓存检索被 MemoryGraph DIALOGUE 节点 + FAISS 侧车索引替代 |
| `SessionVectorCache` | **废弃** | 融入 MemoryGraph 的 FAISS 摘要索引 |
| `EpisodicMemory` | **废弃** | 摘要生成逻辑迁入 EpisodeAdapter 内部，分级读取由图的 BFS 下钻实现 |
| `conversation_history` (内存List) | **废弃** | 被 MemoryGraph DIALOGUE 节点（Round/Sub-dialogue）替代 |
| `ExperienceRAG` | **保留，改为被动** | 取消自动注入，注册为 Function Calling 工具，仅模型主动检索时调用 |
| `KnowledgeRAG` | **保留，打通图谱** | 保留 FAISS 索引能力，通过 KNOWLEDGE 节点的 backend_ref 与图谱联动 |
| `ToolRAG` | **保留** | 与记忆系统无关，属于工具能力索引 |

### 需复用的旧机制

| 机制 | 来源文件 | 复用方式 |
|------|---------|---------|
| `LLMMemoryReviewer` | `llm_memory_reviewer.py` | 集成到 MemoryGraph 剪枝流程，作为"剪枝前 LLM 守卫" |
| `MemoryStrength` | `memory_evolution.py` | importance_level + 衰减公式移植到 GraphNode 标签体系 |
| `MemoryConsolidator` | `memory_evolution.py` | 巩固逻辑适配到标签体系下的重要度提升 |
| `SummaryStore 双索引架构` | `summary_store.py` | FAISS 摘要索引 + 节点指针的分层检索思路融入 MemoryGraph |
| `LLM 审查 Prompt 模板` | `llm_memory_reviewer.py` | PRE_STORE / PRE_EVICT / PERIODIC_REVIEW 审查标准直接复用 |

## 四、任务清单

### 任务 1：GraphNode 扩展——多维标签体系

**目标**：用多组正交标签替代硬分区（L1/L2/L3），实现灵活的记忆分类和检索路由。

**修改文件**：`zulong/memory/memory_graph.py`

**设计说明**：

短期/中期/长期记忆只是模糊的概念，没有硬边界。实际分类通过以下三组正交标签实现：

```
标签维度           存储方式        可选值                           说明
─────────────────────────────────────────────────────────────────────────
温度标签           metadata 存储   hot / warm / cold                基于访问频率+最近访问时间
(temperature)     动态更新        hot: 最近一段时间内被激活过         每次访问/BFS激活时更新
                                 warm: 中等时间未激活
                                 cold: 长期未激活

重要度标签         metadata 存储   trivial / normal / identity /    写入时由规则检测标记
(importance)      写入时设置      fact / important / must_remember  LLM审查/手动可提升
                  可被提升        trivial: 无意义闲聊("嗯"/"好的")
                                 normal: 普通对话
                                 identity: 身份信息(姓名/年龄/称呼)
                                 fact: 客观事实(日期/电话/地址)
                                 important: 承诺/任务指令/偏好
                                 must_remember: 用户显式要求记住

时间段标签         不存储          recent / non_recent              检索时动态计算
(time_scope)      查询时计算      recent: 最近 T 分钟内的节点        T 为可配置参数(默认 30 分钟)
                                 non_recent: 超过 T 分钟的节点
```

**检索路由规则**：

```
用户输入
  ├── 并行路径 A: time_scope=recent 的节点
  │   └── 直接遍历 + BFS（不经过 FAISS）
  │
  └── 并行路径 B: time_scope=non_recent 的节点
      └── FAISS 摘要向量检索 → BFS 下钻
  
  合并: 两路径结果按激活值排序 → Top-N 注入
```

**具体内容**：
1. 在 `GraphNode.metadata` 中定义以下标准字段（通过 metadata 约定，不修改 dataclass）：
   - `temperature`: `"hot"` / `"warm"` / `"cold"`，默认 `"hot"`（新节点刚写入时是热的）
   - `importance`: `"trivial"` / `"normal"` / `"identity"` / `"fact"` / `"important"` / `"must_remember"`，默认 `"normal"`
   - `detected_entities`: `List[Dict]`，自动检测到的实体列表（任务 7 填充）
   - `memory_strength`: 复用 `MemoryStrength.to_dict()` 的衰减追踪结构
2. 温度标签动态更新规则：
   - 节点被 BFS 激活或直接访问时 → 设为 `"hot"`，更新 `last_accessed`
   - 定期检查（在 `decay_and_prune()` 中）：
     - `last_accessed` 在最近 1 小时内 → `"hot"`
     - `last_accessed` 在 1 小时到 24 小时之间 → `"warm"`
     - `last_accessed` 超过 24 小时 → `"cold"`
3. 添加辅助方法：
   - `get_temperature(node_id) -> str`
   - `get_importance(node_id) -> str`
   - `set_importance(node_id, level: str)`
   - `update_temperature(node_id)` — 根据 last_accessed 自动计算
   - `is_recent(node_id, threshold_minutes=30) -> bool` — 检索时判断是否在热窗口内

**验收标准**：
- 新建的节点默认 `temperature="hot"`, `importance="normal"`
- 温度标签在每轮 `decay_and_prune()` 中被自动更新
- `is_recent()` 可正确区分时间窗口内外的节点
- 不影响现有 JSON 持久化/加载

### 任务 2：基于重要度的差异化衰减——decay_and_prune() 改造

**目标**：不同 importance 标签的节点使用不同的衰减半衰期，高重要度节点获得更长的记忆保持时间。

**修改文件**：`zulong/memory/memory_graph.py`

**具体内容**：
1. 修改 `decay_and_prune()` 方法，按 importance 标签选择半衰期：
   ```
   importance         半衰期        说明
   ──────────────────────────────────────────
   must_remember      不衰减        永久保持，边设为 protected
   identity           720h (30天)   身份信息衰减极慢
   fact               360h (15天)   事实性信息衰减慢
   important          168h (7天)    重要信息中等衰减
   normal             24h           普通对话（当前默认值）
   trivial            6h            闲聊快速衰减
   ```
   - 边的衰减半衰期取两端节点中**更高重要度**的值
   - `must_remember` 节点的所有关联边自动设为 `protected=True`
2. 同步更新温度标签（在衰减循环中顺便计算）：
   - 遍历所有节点，根据 `last_accessed` 更新 `temperature`
3. 修改孤立节点清理逻辑：
   - `importance >= "identity"` 的节点永不因孤立而删除
   - `importance == "trivial"` 的孤立节点容忍时间缩短为 6h
   - 其余节点保持当前 24h 容忍时间

**验收标准**：
- `must_remember` 节点的边在任意长时间后权重不变
- `trivial` 节点的边在 6h 后大幅衰减
- `identity` 节点 7 天后仍保持显著权重
- 单元测试覆盖各 importance 级别的衰减行为

---

### 任务 3：重要度动态提升——自动巩固流程

**目标**：根据访问模式和 LLM 审查结果，自动提升节点的 importance 标签，使重要记忆获得更长的生命周期。

**修改文件**：`zulong/memory/memory_graph.py`（新增方法）

**具体内容**：
1. 新增 `promote_importance(node_id, target_importance)` 方法：
   - 仅允许向上提升（不可降级 importance，降级由 LLM 审查专门处理）
   - 提升时如果目标是 `must_remember` → 自动将所有关联边设为 `protected=True`
2. 自动提升规则（新增 `run_importance_review()` 方法，在 `decay_and_prune()` 后调用）：
   - `access_count >= 3` 且当前 `importance == "normal"` → 提升为 `"important"`
   - `access_count >= 5` 且当前 `importance == "important"` → 候选提交 LLM 审查确认是否提升为 `must_remember`
   - 被重要信息检测标记的节点（任务 7）→ 直接设为对应级别（identity/fact/important）
3. LLM 审查触发的提升（与任务 4 联动）：
   - LLM PERIODIC_REVIEW 标记为 PROMOTE → 调用 `promote_importance()` 提升
   - LLM PRE_EVICT 标记为 KEEP → 至少提升一级

**验收标准**：
- 高频访问的 normal 节点自动提升为 important
- must_remember 节点的关联边全部变为 protected
- 提升操作记录在 metadata 中（`importance_history` 列表）

### 任务 4：集成 LLM 剪枝守卫——复用 LLMMemoryReviewer

**目标**：在 MemoryGraph 剪枝流程中集成 LLM 审查，作为"剪枝前最后防线"。

**修改文件**：`zulong/memory/memory_graph.py`、`zulong/memory/llm_memory_reviewer.py`

**具体内容**：
1. 在 `decay_and_prune()` 中，当边权衰减到接近阈值（< 0.15 但 > 0.05）时：
   - 收集这些"濒危"边两端的节点
   - 如果节点 `importance != "trivial"`，提交给 `LLMMemoryReviewer.review_before_evict()`
   - 在审查结果返回前，这些边**暂不剪枝**（设置一个 pending_review 标记）
2. 处理 LLM 审查结果（`process_review_result()` 的回调）：
   - KEEP → 强化边权（reinforce）+ 提升 importance（调用 `promote_importance()`）
   - DISCARD → 允许下一轮剪枝删除
   - COMPRESS → 压缩节点 metadata 中的内容为摘要，重置边权
   - PROMOTE → 调用 `promote_importance()` 提升重要度
   - MERGE → 合并到相关节点（将内容合并到关联节点的 metadata 中）
3. 适配 `LLMMemoryReviewer` 的 Prompt 模板：
   - 修改 `_build_evict_prompt_template()` 使其适配图节点格式（当前是 RAG 文档格式）
   - 添加图上下文：节点的邻居摘要、所属 Session 信息等

**验收标准**：
- 重要节点在衰减到阈值时能被 LLM 审查拦截
- LLM 标记为 PROMOTE 的节点被正确提升 importance
- 审查通过 L2-BACKUP 异步执行，不阻塞主对话流

### 任务 5：FAISS 摘要侧车索引——非热记忆向量检索

**目标**：为 MemoryGraph 建立 FAISS 摘要索引层，支持非热记忆（time_scope=non_recent）的高效向量检索。

**修改文件**：`zulong/memory/memory_graph.py`（新增索引管理）

**说明**：FAISS 索引中**只存摘要向量 + 节点 ID 指针**，不向量化记忆节点的实际内容。记忆节点内容只存在图节点的 metadata 中。

**具体内容**：
1. 新增 `SummaryIndex` 内部类或独立模块：
   - 维护一个 FAISS IndexFlatIP（或 IndexIVFFlat，视规模定）
   - 索引内容：Session 节点的 `session_summary` 摘要向量 + EPISODE 节点的摘要向量
   - 每条向量携带 `node_id` 作为指针
   - 使用与 KnowledgeRAG 相同的 embedding 模型（BAAI/bge-small-zh-v1.5, 512 维）
2. 索引维护：
   - `DialogueAdapter.finalize_round()` 完成时，为当前 Session 节点生成/更新摘要向量并写入 FAISS
   - `EpisodeAdapter.sync()` 时，为 EPISODE 节点生成摘要向量并写入 FAISS
   - 节点温度从 `hot` 变为 `warm` 时，确保其摘要已被索引（热节点不需要索引，因为走遍历路径）
   - 节点删除时，同步移除 FAISS 中的向量
3. 检索接口：
   - `search_by_summary(query_text, top_k=5) -> List[(node_id, score)]`
   - 内部：query_text → embedding → FAISS Top-K → 返回命中节点 ID 和相似度分数
4. 持久化：FAISS 索引随 MemoryGraph 一同保存/加载（`save()` / `_load()`）

**验收标准**：
- FAISS 索引与图节点保持同步（增删一致）
- 1000 条摘要的检索延迟 < 10ms
- 索引随 JSON 一同持久化，重启后可恢复

### 任务 6：并行检索策略——热数据遍历 + 非热数据向量检索

**目标**：在同一张图上，通过标签互斥过滤实现两条并行检索路径，避免重复搜索。

**核心原则**：图是一个整体，不做物理拆分。两条检索路径通过标签过滤在同一张图上各自排除对方的数据范围：
- 全量遍历路径：只扫描热数据，**排除**温/冷数据
- FAISS 向量检索路径：只搜索非热数据，**排除**热数据（热数据已被遍历路径覆盖）

**修改文件**：`zulong/memory/memory_graph.py`（新增检索方法）

**具体内容**：
1. 新增核心检索方法 `async retrieve_context(query_text, top_k=10, hot_window_minutes=30)`：
   ```
   步骤 0: 将当前时间 - hot_window_minutes 作为热窗口边界 T_hot
   
   图是一个整体，两条路径通过标签互斥过滤同一张图：

   并行路径 A: 全量遍历（排除温/冷数据）
   ├── 遍历图中 is_recent(node, hot_window_minutes) = True 的节点
   ├── 直接关键词匹配 + BFS 扩散（量小，通常 <50 个，不需要向量检索）
   └── 返回 Top-K 热记忆节点

   并行路径 B: FAISS 向量检索（排除热数据）
   ├── query_text → embedding → FAISS 摘要索引 Top-K
   ├── FAISS 索引中不包含热数据（热节点不入索引，或检索后过滤掉）
   ├── 命中的 Session/EPISODE 节点作为 BFS 种子，下钻获取详情
   └── 返回 Top-K 非热记忆节点

   合并: 两条路径结果合并（无重复，因为互斥），按激活值排序，返回 Top-N
   ```
2. 路径 A 和路径 B 使用 `asyncio.gather()` 并行执行
3. 合并策略：
   - 热数据有天然优先权（加权 boost，如 1.5x）
   - 高 importance 节点额外加权（identity/fact/important/must_remember 各有 boost 系数）
   - 两条路径结果天然无重复（互斥过滤），直接合并即可
   - 最终 Top-K 按类型格式化为 LLM 可读文本
4. `hot_window_minutes` 为可配置参数，默认 30 分钟，可根据对话频率动态调整

**验收标准**：
- 两条检索路径并行执行，总延迟接近单条路径最长的那一条
- 热记忆（当前对话上下文）优先于冷记忆返回
- 100 节点和 10000 节点场景下的延迟测试

### 任务 7：重要信息自动检测

**目标**：对话写入 DIALOGUE 节点时，自动检测关键实体并标记重要性。

**修改文件**：`zulong/memory/graph_adapters.py`（DialogueAdapter）

**具体内容**：
1. 在 `DialogueAdapter.add_round()` 和 `add_sub_dialogue()` 中，对 user_text 和 bot_text 执行轻量级实体检测：
   - 身份信息检测："我叫"/"我是"/"我姓"/"我今年XX岁" → `importance = "identity"`
   - 事实检测："我家在XX"/"我的电话是"/"XX月XX号" → `importance = "fact"`
   - 偏好检测："我喜欢"/"我不喜欢"/"我爱"/"我讨厌" → `importance = "important"`
   - 承诺检测："我答应"/"我保证"/"帮我记住"/"别忘了" → `importance = "must_remember"`
   - 无意义闲聊检测：内容 < 5 字符或匹配语气词列表（"嗯"/"好"/"哦"） → `importance = "trivial"`
2. 检测到关键信息时：
   - 自动设置节点 `importance` 为对应级别
   - 将检测到的实体写入 metadata 的 `detected_entities` 字段
   - `identity` / `fact` / `important` / `must_remember` 级别的节点自动调用 `promote_importance()`
3. 实现为可配置的规则引擎（便于扩展检测规则）

**验收标准**：
- 用户说"我叫小明"时，该对话节点自动标记为 `importance="identity"`
- 用户说"帮我记住我妈电话是13800138000"时，节点标记为 `importance="must_remember"`
- 普通闲聊（"你好"/"今天天气不错"）保持 `importance="normal"`
- 纯语气词（"嗯"/"好的"）标记为 `importance="trivial"`

### 任务 8：ExperienceRAG 被动化——取消主动注入

**目标**：ExperienceRAG 不再自动注入到 LLM 上下文，改为模型通过 Function Calling 主动检索。

**修改文件**：`zulong/l2/inference_engine.py`、`zulong/l2/rag_node.py`

**具体内容**：
1. 在 `_build_messages_with_history_async()` 中：
   - 移除 `_retrieve_from_rag()` 中对 ExperienceRAG 的自动检索和注入
   - 保留 KnowledgeRAG 的检索路径（任务 9 打通后由图驱动）
2. 将 ExperienceRAG 注册为 Function Calling 工具：
   - 工具名：`search_experience`
   - 参数：`query: str`（检索关键词）
   - 返回：Top-K 经验文档的摘要
   - 注册方式：与其他工具一样通过 ToolRAG 或直接注册到工具列表
3. 经验写入路径保持不变（ExperienceGenerator → ExperienceRAG）

**验收标准**：
- 普通对话中 ExperienceRAG 不再被自动注入到 system prompt
- 模型在需要参考经验时能通过工具调用获取
- 经验生成和存储流程不受影响

### 任务 9：KnowledgeRAG 打通 MemoryGraph

**目标**：MemoryGraph 的检索流程中能通过 KNOWLEDGE 节点反查 KnowledgeRAG 获取完整知识内容。

**修改文件**：`zulong/memory/memory_graph.py`、`zulong/memory/graph_adapters.py`

**具体内容**：
1. 确保 `KnowledgeGraphAdapter.sync()` 正确设置 KNOWLEDGE 节点的 `backend_ref`：
   - 格式：`"knowledge_rag:{doc_id}"` 指向 KnowledgeRAG 中的文档
2. 在 `retrieve_context()`（任务 6）的 BFS 下钻阶段：
   - 当激活到 KNOWLEDGE 类型节点时，检查 `backend_ref`
   - 如果 backend_ref 指向 KnowledgeRAG，通过 RAGManager 反查获取完整文档内容
   - 将完整内容作为检索结果的一部分返回
3. 提供 `resolve_backend_ref(node_id) -> Dict` 通用方法：
   - 解析 backend_ref 格式，路由到对应后端获取完整数据
   - 支持 `knowledge_rag:*`、`experience_rag:*`、`task_graph:*` 等前缀

**验收标准**：

## 六、关键设计决策记录

### 决策 1：向量检索只做摘要索引，不向量化节点内容

FAISS 侧车索引中只存储 **Session / EPISODE 级摘要的向量 + 节点 ID 指针**。记忆节点的实际内容（user_text / bot_text / desc 等）只存在于图节点的 metadata 中，不做向量化。FAISS 命中后，通过 node_id 回到图中，BFS 下钻到子节点获取完整内容。

### 决策 2：一张图 + 多维标签 + 互斥过滤

**记忆图是一个整体，所有节点和边在同一张 NetworkX 图中，不做任何物理拆分。** 标签只是节点上的属性标记，不改变图的拓扑结构，BFS 扩散可以跨任何标签传播。

三组正交标签：
- **温度标签** (`temperature`): hot/warm/cold — 动态计算，反映最近访问频率
- **重要度标签** (`importance`): trivial/normal/identity/fact/important/must_remember — 写入时设置，可被提升
- **时间段标签** (`time_scope`): recent/non_recent — 不存储，检索时按时间窗口动态计算

一个节点可以同时是 `importance="identity"` + `temperature="hot"`（如用户刚提到的名字），既衰减极慢（高重要度），又在热检索路径中被直接遍历到。标签之间不互斥，记忆属性可以自由组合。

### 决策 3：同图互斥过滤检索

两条并行检索路径在**同一张图**上运行，通过标签**互斥过滤**各自排除对方的数据范围，保证无重复：

- **全量遍历路径**：扫描图中 `time_scope=recent` 的节点，**排除**温/冷数据。量小（<50 节点），直接遍历 + BFS + 关键词匹配。
- **FAISS 向量检索路径**：搜索摘要索引，**排除**热数据（热节点已被遍历路径覆盖）。命中摘要后 BFS 下钻到详情节点。
- 两条路径通过 `asyncio.gather()` **并行执行**。因为互斥过滤，结果天然无重复，直接合并。
- T（热窗口）为可配置参数（默认 30 分钟），可根据对话频率动态调整。

### 决策 4：衰减速度由重要度标签决定

不同 `importance` 标签的节点拥有不同的衰减半衰期（从 6h 到永不衰减），而非所有节点统一 24h。这自然实现了"重要信息保持更久"的效果，无需显式的"长期记忆区"概念。`must_remember` 节点的关联边自动设为 `protected`，永不被衰减和剪枝。

### 决策 5：LLM 审查是异步守卫，不阻塞主对话

所有 LLM 审查（PRE_EVICT / PERIODIC_REVIEW / importance 提升确认）通过 L2-BACKUP 调度器异步执行。如果审查结果尚未返回，濒危节点暂不剪枝（设置 pending_review 标记），等审查完成后再决定。
