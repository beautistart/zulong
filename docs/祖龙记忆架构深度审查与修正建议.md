# 祖龙记忆架构深度审查与修正建议

> 审查日期: 2026-05-01
> 基于代码版本: zulong_beta4
> 审查范围: 记忆图谱(MemoryGraph)、任务图谱(TaskGraph)、注意力窗口(AttentionWindow)、FC工具链、持久化机制
> 已实施修正: P2-11, P1-10, P1-3, P2-7, P2-8 (共5项)

---

## 一、设计哲学确认

祖龙记忆系统**模仿人类神经认知结构**：

| 人脑结构 | 祖龙对应组件 | 职责 |
|----------|-------------|------|
| 前额叶皮层 (意识决策) | LLM + FC 工具 (recall_memory, save_memory_note, navigate_attention 等) | 有意识的记忆检索、注意力导航 |
| 海马体 (记忆形成/检索) | `retrieve_context()` + `save_memory_note` | 新记忆编码、上下文检索 |
| 突触可塑性 (赫布学习) | `hebbian_strengthen()` + `_update_coactivation_counter()` | 共激活关联自动强化 |
| 网状激活系统 (唤醒扩散) | `compute_activations()` BFS 扩散 | 从焦点节点向外扩散激活 |
| 遗忘曲线 | `decay_and_prune()` 艾宾浩斯衰减 | 基于重要性的半衰期遗忘 |
| 注意力网络 (DMN/TPN切换) | `AttentionWindow` 三模式 (GLOBAL/FOCUS/SINGLE_CHAIN) | 动态注意力焦点调节 |

**核心原则**: 意识层(LLM)通过FC工具交互，无意识层(BFS/赫布/衰减)自动运行。两层协同但不相互暴露底层细节。

## 四、9项设计对齐的修正建议

### P0 级 (关键缺陷)

#### 修正1: 工作记忆在焦点漂移时不刷新

**问题**: `_maybe_drift_focus()` (scheduler_gatekeeper.py:117) 切换焦点节点后，系统 prompt 中的 `【记忆上下文】` 不会重新检索。LLM 在新焦点下仍然看到旧焦点的记忆上下文。

**类比**: 人把注意力从"写代码"转到"回复邮件"时，工作记忆应该自动更新为邮件相关的记忆，而不是继续留着代码片段。

**代码影响**:
- inference_engine.py `_build_chat_messages()` / `_build_complex_messages()` 只在构建时调用一次 `retrieve_context()`
- 焦点漂移发生在 FC 循环中间，但系统 prompt 已经固化

**修正**: 焦点漂移后，标记 "memory_context_stale=True"，在下次 `call_model_node` 前重新执行 `retrieve_context()` 并替换系统 prompt 中的记忆段。

#### 修正2: ~~共激活计数器未持久化~~ (已确认为误判)

代码审查确认: `_coactivation_counter` 已在 `save()` (memory_graph.py:2174-2177) 中序列化，在 `_load()` (memory_graph.py:2259-2262) 中反序列化。此项不需要修正。

### P1 级 (功能增强)

#### 修正3: BFS 种子扩展到检索命中节点

**问题**: fc_graph.py:418-485 中 BFS 种子仅取自 TaskGraph 的 in_progress 节点。检索命中的记忆节点不参与 BFS 扩散。

**类比**: 人回忆起一件事(检索命中)后，这件事会自动激活相关联想(BFS)。当前实现中，只有"正在做的任务"能触发联想。

**修正**: 将 `retrieve_context()` 返回的 top-3 节点 ID 也加入 BFS seed_node_ids。

#### 修正4: 注意力模式自动回退

**问题**: AttentionWindow 的模式切换是单向的 (GLOBAL→FOCUS→SINGLE_CHAIN)，缺少自动回退机制。如果任务完成或被挂起，注意力应自动回到 GLOBAL。

**修正**: 在 TaskNode 状态变为 completed/suspended 时，触发模式回退。

#### 修正5: navigate_attention 触发无意识联想

**问题**: `navigate_attention` (attention_tool.py) 调用 `mg.update_focus_to_node()` 后，没有触发 BFS 扩散或记忆重检索。注意力"跳转"了但"看不到"新位置的关联记忆。

**修正**: navigate_attention 执行后，自动触发一次:
1. `compute_activations([new_focus_node])` — BFS扩散
2. `retrieve_context(node.label)` — 记忆重检索
3. 标记 memory_context_stale — 触发系统 prompt 刷新

### P2 级 (体验优化)

#### 修正6: 已完成任务渐进衰减

**问题**: 任务完成后立即从焦点路径移除，但对应的 MemoryGraph 节点激活值仍然较高，可能干扰后续注意力分配。

**修正**: 对已完成任务的 MemoryGraph 投影节点，每个衰减周期额外施加 0.3 的衰减系数。

#### 修正7: save_memory_note 自动语义关联

**问题**: `save_memory_note` 创建的 KNOWLEDGE 节点只建立一条 REFERENCE 边到当前焦点任务节点，不做语义关联。如果该笔记与其他记忆有语义相似性，不会自动建边。

**修正**: save_memory_note 后自动调用 `discover_semantic_neighbors()` (memory_graph.py:1635)，对相似度 > 0.75 的节点自动创建 ASSOCIATION 边。

#### 修正8: 工作记忆共激活赫布通路

**问题**: 赫布学习只在 `compute_activations()` 的 BFS 路径上记录 activated_edges，而 retrieve_context 返回的多个并列结果之间不会产生共激活。

**类比**: 人同时回忆起两件事时，即使它们原本没关联，也会因为"同时出现在意识中"而建立弱关联。

**修正**: retrieve_context 返回的 top-K 结果节点，两两记录一次 coactivation。

### P3 级 (长期优化)

#### 修正9: 热路径增加向量语义检索

**问题**: `_retrieve_hot()` 只用关键词匹配(子串+bigram)，无向量语义。热数据中语义相关但关键词不重叠的节点无法命中。

**修正**: 利用已有的 `_embeddings` dict，对热窗口内节点做 numpy 余弦排序，与关键词得分加权合并。

## 五、五大专项分析

### 5.1 记忆检索：关键词+语义并行执行分析

**当前架构** (memory_graph.py:1714-1955):

```
retrieve_context(query_text, top_k, hot_window_minutes, session_id)
  ├─ asyncio.gather()
  │   ├─ _retrieve_hot()   [路径A: 热数据, 纯关键词匹配]
  │   │   ├─ 子串匹配 (score=0.8)
  │   │   ├─ bigram overlap (score=0~0.6)
  │   │   ├─ session_id boost (+0.3)
  │   │   ├─ focus_path boost (+0.2)
  │   │   ├─ importance boost (+0.15~0.3)
  │   │   └─ BFS 1-hop 扩展 (仅热节点)
  │   │
  │   └─ _retrieve_cold()  [路径B: 冷数据, 纯FAISS向量检索]
  │       ├─ FAISS search (排除热节点)
  │       ├─ importance boost (+0.1~0.2)
  │       ├─ focus_path boost (+0.15)
  │       └─ BFS 下钻获取子节点详情
  │
  └─ merge + dedup + sort by score → top_k
```

**问题**: 两路径按温度分工而非按方法分工，导致:
- 热数据: 只有关键词匹配，语义相近但文字不同的热节点无法命中
- 冷数据: 只有向量语义，精确关键词(如错误代码、函数名)可能被语义泛化稀释

**修正方案**:

```
retrieve_context(query_text, top_k, hot_window_minutes, session_id)
  ├─ asyncio.gather()
  │   ├─ _retrieve_hot()
  │   │   ├─ 关键词匹配 (现有逻辑)
  │   │   └─ [新增] 热节点 embedding 余弦排序 (用 _embeddings dict, 不需 FAISS)
  │   │       → 合并: final_score = 0.6*keyword_score + 0.4*semantic_score
  │   │
  │   └─ _retrieve_cold()
  │       ├─ FAISS 向量检索 (现有逻辑)
  │       └─ [新增] 关键词回退: 对 FAISS 未命中的冷节点做子串扫描
  │           → 精确匹配 score=0.7 (确保函数名/错误码等确切字符串不丢失)
  │
  └─ merge + dedup + sort by score → top_k
```

### 5.2 原子写入与持久化审计

#### MemoryGraph 持久化 (memory_graph.py:2156-2290)

**写入流程** (memory_graph.py:2156):
```
save():
  1. 构建完整 data dict (nodes, edges, embeddings, coactivation_counter, meta)
  2. 写入 temp_filepath (.tmp)
  3. 备份当前文件到 .bak
  4. os.replace(temp_filepath, filepath)   ← 原子操作 (NTFS/POSIX)
  5. 保存 FAISS sidecar (独立于JSON)
  6. 标记 _dirty=False
```

**自动保存** (memory_graph.py:2131):
```
_mark_dirty():
  取消上一个定时器 → 新建 2s 延迟 daemon Timer → 回调 _do_auto_save()
```

**崩溃恢复** (memory_graph.py:2221):
```
_load():
  尝试主文件 memory_graph.json → 失败则尝试 .bak → 恢复 nodes/edges/embeddings/coactivation/stats/focus_context
```

**评级**: **优秀** — 完整的原子写入 + 自动保存 + 崩溃恢复链路。

#### TaskGraph 持久化 (task_tools.py:271-309)

```
_backup_graph_to_disk():
  1. tg.serialize() → dict
  2. 写入 .tmp 文件 (带 f.flush() + os.fsync())
  3. os.replace(tmp, backup)
```

**问题**:
- 无 `_mark_dirty()` 自动保存机制 — 只在工具调用后手动 `_save_active_backup()`
- 无 `.bak` 备份文件 — 崩溃时只有最后一次手动备份
- 节点状态变更 (pending→in_progress→completed) 不会自动触发落盘

**风险场景**: LLM 将任务节点标记为 in_progress → 系统崩溃 → 重启后该节点仍为 pending → 重复执行

**修正建议**: 为 TaskGraph 增加与 MemoryGraph 相同的防抖自动保存:
- `TaskNode.status` setter 中调用 `_mark_dirty()`
- 2s 延迟后自动 `_backup_graph_to_disk()`
- 保留上一版本为 `.bak`

#### FAISS Sidecar 持久化 (memory_graph.py:2204)

**问题**: `_summary_index.save()` 直接写文件，无原子保护。

**修正**: 改为先写 `.tmp` 再 `os.replace()`。

### 5.3 地址继承系统分析

#### 当前地址格式

```
TaskGraph 节点:
  task:tg_abc123                    ← TaskGraph 根节点 (MemoryGraph 中的投影ID)
  task:tg_abc123/node_1             ← 一级子任务
  task:tg_abc123/node_1/node_1_1    ← 二级子任务
  dialogue:session_xxx/task:tg_abc  ← 有会话上下文时的完整路径

Dialogue 节点:
  dialogue:session_20240101_120000  ← 会话节点
  dialogue:session_xxx/round_1      ← 对话轮次

Knowledge 节点:
  knowledge:note_{uuid}             ← 知识笔记

File 节点:
  file:{file_path}                  ← 文件引用
```

#### 物理存储映射

| 抽象地址 | 物理位置 |
|----------|----------|
| 所有 GraphNode | `./data/memory_graph/memory_graph.json` 中的 `nodes.{node_id}` |
| TaskGraph 备份 | `./data/graph_backups/{graph_id}.json` |
| FAISS 向量 | `./data/memory_graph/summary_sidecar/` |
| 经验数据 | RAGManager 管理的向量库 |

#### 评估结论

**抽象地址是正确的设计选择**:

1. 图结构本质是拓扑关系，不适合树形物理路径
2. 一个节点可被多个父节点引用（FILE节点同时关联多个TASK），物理文件系统无法直接表达
3. 类比人脑：神经元"地址"是突触拓扑，不是物理坐标
4. `resolve_address()` + `resolve_backend_ref()` 已实现地址→数据的完整解析链路

**改进建议**: 统一地址格式为规范 URI: `{type}:{scope}/{id}`，减少 resolve_address() 中的多重 fallback 匹配。

### 5.4 是否应暴露 BFS/赫布/list_memory 工具

#### 分层对照

| 暴露工具 | 对应人脑操作 | 人类能否主动执行 | 暴露后风险 |
|----------|-------------|-----------------|-----------|
| compute_activations (BFS) | 控制突触激活传播 | 不能 | LLM缺乏图拓扑全局视角，选种子节点盲目 |
| hebbian_strengthen | 调节突触权重 | 不能 | LLM幻觉可能导致错误关联被强化 |
| list_memory (浏览全图) | SELECT * FROM 大脑 | 不能 | 数万节点token溢出上下文窗口 |
| decay_and_prune | 控制遗忘速率 | 不能 | LLM偏向"保留一切"，破坏遗忘机制 |

#### 结论

**不应直接暴露底层机制，但应增强"元认知"接口**:

1. **记忆置信度反馈**: recall_memory 返回时附带 coverage_hint（"图谱中还有N个高关联节点未展示"）
2. **定向深挖**: discover_related(depth=2) 底层自动触发 BFS+赫布，LLM只看结果
3. **记忆标注**: 允许LLM标注"这两条记忆相关"，系统转化为 coactivation 增加
4. **遗忘保护**: 允许LLM标注"这条记忆很重要请不要遗忘"，系统转化为 promote_importance()

这种方式超越人脑(人不能主动要求"深度联想")，同时不破坏分层架构。

## 八、总结

祖龙记忆架构的核心设计——模仿人类神经认知的意识/无意识分层——是独特且正确的。主要问题集中在:

1. **检索能力不对称**: 热路径缺向量语义，冷路径缺关键词精确匹配
2. **工作记忆不刷新**: 焦点漂移后系统prompt中的记忆上下文不更新(P0级)
3. **TaskGraph持久化薄弱**: 无自动保存，崩溃可能丢失任务状态
4. **注意力导航"断联"**: navigate_attention不触发BFS/检索，跳转后看不到新位置的关联记忆

这些修正都遵循"增强无意识层自动响应性"的原则，而非"给意识层更多控制权"，与祖龙模仿人脑认知的设计哲学保持一致。
