## 一、注意力切换的两种驱动机制

祖龙的注意力机制采用**双重驱动**设计，两种机制互补工作：

```
┌─────────────────────────────────────────────────────────────────┐
│                    注意力模式切换决策                             │
└─────────────────────────────────────────────────────────────────┘
                              │
              ┌───────────────┴───────────────┐
              │                               │
              ▼                               ▼
┌─────────────────────────┐     ┌─────────────────────────┐
│   机制A: 工具驱动切换     │     │   机制B: 压力驱动切换     │
│   (确定性状态机)         │     │   (LLM自主选择)          │
└─────────────────────────┘     └─────────────────────────┘
              │                               │
              │                               │
              ▼                               ▼
┌─────────────────────────┐     ┌─────────────────────────┐
│ • 无需压力阈值           │     │ • 压力值超过阈值时触发    │
│ • 工具类型决定目标模式    │     │ • LLM决策切换到什么模式   │
│ • 立即生效               │     │ • 有冷却期防止震荡        │
│ • 确定性规则             │     │ • 设计之内的自主选择      │
└─────────────────────────┘     └─────────────────────────┘
```

## 二、机制A: 工具驱动切换（确定性状态机）

### 2.1 触发流程

```
LLM推理 → 生成工具调用 → observe_tool_call() → 模式切换
```

**关键代码位置**: `attention_window.py:240-285`

```python
def observe_tool_call(
    self,
    tool_name: str,
    tool_args: Dict,
) -> Optional[AttentionMode]:
    """观察工具调用，驱动模式状态机"""
    
    old_mode = self.mode
    new_mode = self._compute_transition(tool_name, tool_args)  # ← 核心决策
    
    if new_mode and new_mode != old_mode:
        self.mode = new_mode
        logger.info(
            f"[AttentionWindow] 模式切换: {old_mode.value} → {new_mode.value} "
            f"(触发: {tool_name}, node={self._current_node_id})"
        )
        # 聚焦切换时，自动注入目标节点的已有知识
        if new_mode in (AttentionMode.FOCUS, AttentionMode.SINGLE_CHAIN):
            self._inject_node_knowledge(self._current_node_id)
        return new_mode
    
    return None
```

# GLOBAL → FOCUS 触发器（聚焦操作）
_FOCUS_TRIGGERS: Set[str] = {
    "recall_memory",        # 检索记忆 → 需要聚焦相关内容
    "read_memory_node",     # 读取记忆节点 → 需要聚焦该节点

    "search_experience",    # 搜索经验 → 需要聚焦相关经验
    "task_get_detail",      # 查看任务详情 → 需要聚焦该任务

    "exec_run_command",     # 执行命令 → 进入单链执行模式
}

_GLOBAL_FORCE_TRIGGERS: Set[str] = {
    "task_view_overview",   # 查看任务概览 → 回到全局视角
    "submit_final_answer",  # 提交最终答案 → 回到全局视角

### 2.3 状态机决策逻辑

**代码位置**: `attention_window.py:403-447`

```python
def _compute_transition(self, tool_name: str, tool_args: Dict) -> Optional[AttentionMode]:
    """根据工具名和参数计算目标模式"""
    
    # 1. navigate_attention 由 on_navigate_attention 单独处理
    if tool_name == "navigate_attention":
        return None
    
    # 2. adjust_attention_mode: LLM直接指定目标模式（显式控制）
    if tool_name == "adjust_attention_mode":
        mode_str = tool_args.get("mode", "")
        _mode_map = {
            "global": AttentionMode.GLOBAL,
            "focus": AttentionMode.FOCUS,
            "single_chain": AttentionMode.SINGLE_CHAIN,
        }
        return _mode_map.get(mode_str)  # LLM想要什么模式就给什么
    
    # 3. 强制回全局
    if tool_name in _GLOBAL_FORCE_TRIGGERS:
        return AttentionMode.GLOBAL
    
    # 4. task_mark_status 根据 status 参数决定
    if tool_name == "task_mark_status":
        status = tool_args.get("status", "")
        if status in ("completed", "skipped"):
            return AttentionMode.GLOBAL  # 任务完成 → 回到全局
        if status == "in_progress":
            if self.mode == AttentionMode.FOCUS:
                return AttentionMode.SINGLE_CHAIN  # 开始执行 → 单链模式
            return None
    
    # 5. GLOBAL → FOCUS（聚焦操作）
    if self.mode == AttentionMode.GLOBAL:
        if tool_name in _FOCUS_TRIGGERS:
            return AttentionMode.FOCUS
    
    # 6. FOCUS → SINGLE_CHAIN（执行操作）
    if self.mode == AttentionMode.FOCUS:
        if tool_name in _SINGLE_CHAIN_TRIGGERS:
            return AttentionMode.SINGLE_CHAIN
    
    return None  # 保持当前模式
```

def apply_window(self) -> List[Dict]:
    """对所有消息评分、裁剪，返回适合 LLM 的消息列表"""
    
    # ── LLM自主注意力选择检查 ──
    if self._should_try_llm_selection():  # ← 检查是否应该尝试LLM选择
        try:
            import asyncio
            loop = asyncio.get_event_loop()
            if loop.is_running():
                asyncio.create_task(self._try_llm_mode_selection_async())
            else:
                loop.run_until_complete(self._try_llm_mode_selection_async())

            logger.warning(f"[AttentionWindow] LLM注意力选择执行失败: {e}")

### 3.2 压力检测与阈值

**代码位置**: `zulong/l2/pressure_detector.py:41-80`

```python
def calculate_pressure(self) -> PressureMetrics:
    """计算当前上下文压力"""
    
    total_tokens = sum(env.tokens for env in self._awm.envelopes)
    current_budget = self._awm.budget
    
    # 压力值 = 当前token / 预算
    pressure_value = total_tokens / current_budget
    
    # 预算 = (上下文窗口 - 预留) × 90%
    # 例如 DeepSeek-V4: budget = (131072 - 7096) × 0.9 = 111578
    
    velocity = self._calculate_velocity()  # 压力变化速度
    predicted = self._predict_pressure(pressure_value, velocity)  # 预测5秒后压力
    trend = self._determine_trend(pressure_value)  # 趋势：上升/下降/稳定
    
    return PressureMetrics(
        current_pressure=pressure_value,
        pressure_trend=trend,
        pressure_velocity=velocity,
        predicted_pressure_5s=predicted,
        budget_usage=budget_usage,
        message_count=message_count,
        token_density=token_density,
    )
```

### 3.3 压力阈值是动态计算的

**关键配置**: `zulong/l2/attention_config.py`

```python
class AttentionConfig:
    """LLM自主注意力选择配置"""
    
    # 压力阈值（触发LLM选择的条件）
    pressure_threshold_high: float = 0.85  # 高压阈值
    pressure_threshold_low: float = 0.60   # 低压阈值
    
    # 这些阈值会根据模型上下文窗口动态调整：
    # - 小窗口模型（如GPT-3.5 4K）：阈值会更低，更早触发
    # - 大窗口模型（如DeepSeek-V4 131K）：阈值保持默认
```

**动态调整逻辑**（推断）：

```python
# 压力阈值可能根据上下文窗口大小动态调整
def get_pressure_threshold(context_window_size: int) -> float:
    if context_window_size < 10000:
        # 小窗口：更激进，阈值更低
        return 0.70
    elif context_window_size < 50000:
        # 中等窗口：默认阈值
        return 0.85
    else:
        # 大窗口：保持默认，或可设置更高
        return 0.85
```

### 3.4 LLM自主选择流程

```
压力值 > 阈值
    │
    ▼
_should_try_llm_selection() 返回 True
    │
    ▼
_try_llm_mode_selection_async()
    │
    ├─ 构建决策请求：当前压力、候选模式、上下文摘要
    │
    ├─ 调用 L2-BACKUP 模型进行决策
    │   Prompt: "根据当前压力情况，建议切换到什么注意力模式？"
    │
    ├─ 解析LLM响应：决策 + 理由
    │
    └─ 执行切换：
        • 检查冷却期（防止震荡）
        • 检查震荡检测器
        • 执行模式切换
        • 记录切换原因："LLM自主选择"
```

### 3.5 设计目标

**机制B的设计意图**：

| 场景 | 机制A能否处理 | 机制B的作用 |
|------|--------------|------------|
| LLM调用recall_memory | ✅ 自动切换到FOCUS | 不需要 |
| 压力快超限，但LLM未调用聚焦工具 | ❌ 无法处理 | ✅ LLM主动决策切换 |
| 长时间执行，上下文累积 | ❌ 无法处理 | ✅ LLM根据压力主动调整 |
| 复杂任务，需要动态调整注意力 | 部分 | ✅ 补充机制A的不足 |

**关键点**：
- **这是设计之内的LLM自主选择**，不是bug
- **压力阈值是动态计算的**，不会对大窗口模型产生负面影响
- **有冷却期和震荡检测器**，防止频繁切换

## 四、为什么大窗口模型下"工具驱动更明显"

### 4.1 实际表现

| 模型 | 上下文窗口 | Budget | 典型对话Tokens | 压力值 | 阈值 | 是否触发机制B |
|------|-----------|--------|---------------|--------|------|--------------|
| GPT-3.5 | 4,096 | 2,800 | 2,500 | 0.89 | 0.85 | ✅ 触发 |
| GPT-4 | 8,192 | 5,900 | 5,000 | 0.85 | 0.85 | ✅ 边界 |
| Claude-3 | 100,000 | 83,300 | 30,000 | 0.36 | 0.85 | ❌ 不触发 |
| DeepSeek-V4 | 131,072 | 111,578 | 50,000 | 0.45 | 0.85 | ❌ 不触发 |

**结论**：
- 大窗口模型下，压力值长期低于阈值，**机制B很少触发**
- 但**机制A照常工作**，工具驱动的切换不受影响

### 4.2 两种机制的关系

```
┌─────────────────────────────────────────────────────────────────┐
│                    大窗口模型（如DeepSeek-V4）                    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼
         ┌────────────────────────────────────────┐
         │  压力值 ≈ 0.3-0.5 < 阈值(0.85)          │
         │  → 机制B（压力驱动）几乎不触发           │
         └────────────────────────────────────────┘
                              │
                              ▼
         ┌────────────────────────────────────────┐
         │  但机制A（工具驱动）照常工作！           │
         │  → LLM调用recall_memory → FOCUS        │
         │  → LLM调用exec_write_file → SINGLE     │
         │  → LLM调用submit_final_answer → GLOBAL │
         └────────────────────────────────────────┘
                              │
                              ▼
         ┌────────────────────────────────────────┐
         │  结果：模式切换完全由工具类型决定        │
         │        （但这是正常行为，不是bug）       │
         └────────────────────────────────────────┘
```

**关键理解**：
- **机制A不依赖压力**，所以不受窗口大小影响
- **机制B依赖压力**，所以大窗口下很少触发
- **两者互补**，机制A覆盖大部分场景，机制B作为安全网

## 五、LLM如何通过工具控制注意力

### 5.1 显式控制：adjust_attention_mode 工具

**LLM可以直接指定模式**：

```json
{
  "tool_calls": [{
    "function": {
      "name": "adjust_attention_mode",
      "arguments": "{\"mode\": \"focus\"}"
    }
  }]
}
```

**效果**：立即切换到FOCUS模式，无需压力触发

### 5.2 隐式控制：通过工具选择

**LLM可以通过选择工具隐式切换模式**：

```
LLM内部决策: "我需要检索记忆"
    │
    ▼
LLM调用: recall_memory
    │
    ▼
observe_tool_call("recall_memory", ...)
    │
    ▼
_compute_transition() 返回 FOCUS
    │
    ▼
模式切换: GLOBAL → FOCUS
```

**这就是"工具驱动"的本质**：
- LLM选择工具时，隐含地表达了"我需要什么模式"
- 系统根据工具类型自动切换，无需LLM额外决策

### 5.3 navigate_attention 工具

**LLM可以显式导航注意力焦点**：

```json
{
  "tool_calls": [{
    "function": {
      "name": "navigate_attention",
      "arguments": "{\"direction\": \"deeper\"}"
    }
  }]
}
```

**效果**：
- `deeper`: GLOBAL → FOCUS → SINGLE_CHAIN
- `broader`: SINGLE_CHAIN → FOCUS → GLOBAL
- `jump`: 根据目标节点深度自动选择

## 六、总结：双重驱动机制的价值

| 维度 | 机制A（工具驱动） | 机制B（压力驱动） |
|------|------------------|------------------|
| **触发条件** | LLM调用特定工具 | 压力值超过阈值 |
| **决策者** | 确定性状态机 | LLM自主选择 |
| **延迟** | 立即生效 | 有冷却期 |
| **适用场景** | 大部分常规操作 | 长时间执行、压力过高 |
| **受窗口大小影响** | 否 | 是 |
| **设计目标** | 让LLM通过工具隐式控制 | 安全网、补充机制A不足 |

**核心设计理念**：
1. **机制A是主力**：覆盖90%的场景，无需压力检测
2. **机制B是安全网**：覆盖机制A无法处理的场景（如压力过高但LLM未调用聚焦工具）
3. **两者互补**：不会因为大窗口导致注意力机制失效

**对于大窗口模型**：
- 机制A照常工作，LLM通过工具选择控制注意力
- 机制B很少触发，但这是正常现象，不是问题
- 如果需要更早触发机制B，可以降低压力阈值（但通常不需要）
