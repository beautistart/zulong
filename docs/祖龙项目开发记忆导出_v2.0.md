## 一、项目核心记忆

### 1.1 项目定位与愿景

**祖龙 (ZULONG)** 是一个面向AI机器人的生产级多层次自适应智能体系统，提供模型推理服务与可观测性一体化能力。

**核心愿景**: 
- 打造"认知科学驱动的系统级AI"，而非单纯依赖模型能力
- 实现"系统承担认知基础设施，模型只负责局部决策"的架构哲学
- 支持端侧小模型到云端大模型的全场景部署

### 1.2 整体架构 (四层模型)

```
┌─────────────────────────────────────────────────────┐
│  L3 专家层: 多专家协作 (可本地/云端)                 │
│  - 逻辑推理/创意生成/视觉处理/TTS/导航/操作         │
│  - 支持动态加载/卸载、热切换 (<10ms)                │
├─────────────────────────────────────────────────────┤
│  L2 认知层: 核心大脑 (唯一决策者)                    │
│  - FC循环 (Function Calling 自主迭代)               │
│  - 任务图谱 (TaskGraph) + 记忆图谱 (MemoryGraph)    │
│  - 注意力窗口管理 + 信息缺口检测                    │
│  - 支持云端大模型/vLLM/本地模型                     │
─────────────────────────────────────────────────────┤
│  L1-B 调度层: 交通指挥                              │
│  - 事件优先级路由 (CRITICAL > HIGH > NORMAL > LOW)  │
│  - 注意力控制器 + 中断管理                          │
├─────────────────────────────────────────────────────┤
│  L1-A/C 感知层: 眼睛和耳朵                          │
│  - 视觉检测 (YOLOv10 + MediaPipe)                   │
│  - 音频处理 (VAD + MFCC)                            │
├─────────────────────────────────────────────────────┤
│  L0 传感器层: 硬件抽象                              │
│  - USB摄像头/麦克风/扬声器驱动                       │
└─────────────────────────────────────────────────────┘
```

## 二、架构决策记忆

### 2.1 核心架构哲学

**决策**: "集中认知 + 分布式技能" 而非 "多Agent水平协作"

**理由**:
1. **一致性保证**: L2唯一决策者，天然避免目标冲突
2. **资源效率**: 共享状态，零同步开销
3. **调试简单**: 单一决策链路，易于追踪
4. **端侧友好**: 适合RTX 3060 6GB等资源受限场景
5. **模型无关**: 同一架构可适配本地小模型/云端大模型

**对比市面多Agent**:
- AutoGen/CrewAI: 分布式决策，水平协商，高资源消耗
- 祖龙: 集中决策，垂直调用，灵活部署

### 2.2 模型部署策略

**决策**: 支持全场景部署模式

**四种部署模式**:
1. **纯本地**: 4B-8B小模型，零API成本，适合离线/隐私场景
2. **纯云端**: GPT-4/Claude等，认知能力无上限
3. **混合模式**: L2云端决策 + L3本地执行，平衡成本与性能
4. **vLLM模式**: 本地vLLM服务，高并发低延迟

**配置示例** (`zulong_config.yaml`):
```yaml
l2_inference:
  core_model: qwen3.5:cloud      # 云端模型
  backup_model: qwen3.5:cloud    # 备用云端模型
```

### 2.3 记忆系统设计

**决策**: 认知科学驱动的异构图记忆

**核心特性**:
- **9种节点类型**: TASK, DIALOGUE, KNOWLEDGE, EXPERIENCE, EPISODE, FILE, CONCEPT, PERSON, DOCUMENT
- **7种边类型**: HIERARCHY, DEPENDENCY, REFERENCE, TEMPORAL, SEMANTIC, CAUSAL, ASSOCIATION
- **赫布学习**: `new_w = old_w + 0.1 × (1 - old_w)`，渐近饱和
- **艾宾浩斯衰减**: 6级重要度半衰期 (TRIVIAL 6h → MUST_REMEMBER ∞)
- **三维标签**: Temperature/Importance/TimeScope 正交组合
- **BFS扩散激活**: graph_boost最大2.0x加成

**市场竞争力**: 全市场独有的"记忆+任务+代码三角闭环"

### 2.4 任务编排设计

**决策**: TaskGraph无限深度递归树 + LangGraph编排

**核心机制**:
- 拓扑排序自动调度
- 状态自动级联 (子节点完成→父节点更新)
- 挂起/恢复完整持久化
- 与MemoryGraph双向同步

### 2.5 拟人化主动反应架构 (NEW)

**决策**: 三层动机驱动模型 (非预设式主动)

**架构设计**:
```
Layer 3: 主动表达层（输出）
  - 生成自然语言
  - 把握语气、时机、长度

Layer 2: 动机评估层（决策）
  - 当前是否有"说话冲动"？
  - 哪种动机最强？（好奇/关心/分享...）
  - 现在说话是否合适？

Layer 1: 动机生成层（感知）
  - 记忆图谱激活值监测
  - 对话上下文语义分析
  - 情感状态追踪
```

**核心原则**:
- ✅ 内在动机驱动 (好奇心、关心、分享欲)
- ✅ 情境感知能力 (识别自然断点、用户状态)
- ✅ 基于记忆图谱的联想能力
- ✅ 主动价值判断 (判断此时说话是否有意义)
- ✅ 精准时机把握 (知道何时该说、何时该沉默)

**vs 预设式主动**:
- ❌ 定时触发、话题库抽取 → 机械执行
- ✅ 动机生成→评估→表达 → 拟人化反应

## 三、开发经验与教训

### 3.1 FC循环防护经验

**5层防护链** (约500行代码，系统最复杂节点):

1. **CB强制收敛**: Circuit Breaker RED状态时强制文本输出
2. **RuleGuardian拦截**: 检查TaskGraph未完成节点，防止过早声明完成
3. **InfoGap信息缺口检测**: 检测NEED_SUBTASK_RESULT/NEED_USER_INPUT
4. **RESUME AutoMark**: 4B模型忘记调task_mark_status时自动补标
5. **COMPLEX Backfill**: 从回复中匹配节点标签，自动回填completed

**教训**: 
- ❌ 早期版本无防护链，4B模型经常失控循环
- ✅ 5层防护链将4B模型完成率从5-15%提升到55-65% (中等复杂度任务)

### 3.2 模块联动问题修复经验

**v3.0修复的6个联动bug**:

| 问题 | 影响 | 修复方案 |
|------|------|---------|
| 已完成任务追问误创新图谱 | 用户追问"怎么运行"创建新任务 | 三层防御 (预防+拦截+保障) |
| 父节点状态不级联 | 子节点完成父节点仍pending | 自动级联递归向上 |
| MemoryGraph激活值不同步 | 旧任务干扰检索 | status→activation映射 |
| 边衰减last_activated缺失 | 边永不衰减 | fallback到created_at |
| 工具结果未截断 | token预算耗尽 | 注册时防御性截断 |
| Pinned消息超限 | 对话历史丢失 | 渐进式降级策略 |

**教训**: 模块间耦合必须显式管理，隐式依赖会导致难以追踪的bug

### 3.3 小模型补偿机制

**8种专项补偿** (针对4B模型):

| 机制 | 解决问题 | 效果 |
|------|---------|------|
| Backfill安全网 | 创建骨架后直接输出 | 自动回填节点 |
| RESUME AutoMark | 忘记调task_mark_status | 自动标记完成 |
| RuleGuardian | 过早声明完成 | 拦截并纠正 |
| CB模式放宽 | 恢复模式需要更多步数 | pattern_window 6→20 |
| null_response限流 | 多层拦截交叉循环 | 连续3次null强制END |
| 强制收敛注入 | 模型陷入循环 | null_count≥2移除工具 |
| 空回复降级 | CB RED后返回空字符串 | 工具结果缓冲区组装 |
| 填充内容检测 | "正在思考"等无实质回复 | 模式匹配拒绝 |

**核心认知**: 系统能防止模型失控，但不能替代模型思考

### 3.4 节点审查任务中断机制 (NEW)

**问题**: 系统触发节点审查后，任务无法被打断，即使有新任务输入也会继续执行占用L2资源

**修复方案** (`zulong/l2/backup_scheduler.py`):

```python
# 新增中断控制
self._current_task: Optional[asyncio.Task] = None
self._interrupt_flag = False

def interrupt_current_task(self):
    """中断当前正在执行的任务"""
    if self._current_task and not self._current_task.done():
        self._interrupt_flag = True
        self._current_task.cancel()

# L2调用支持中断检查
async def _call_l2_backup_with_interrupt_check(self, conversation_turns):
    task = asyncio.create_task(self._call_l2_backup(conversation_turns))
    
    while not task.done():
        if self._interrupt_flag:
            task.cancel()
            raise asyncio.CancelledError("任务被中断")
        await asyncio.sleep(0.1)
    
    return await task
```

**效果**:
- ✅ 用户发起新任务时可中断审查任务
- ✅ 释放L2资源，提升响应速度
- ✅ 新增中断统计 `total_tasks_interrupted`

**教训**: 后台异步任务必须有中断机制，否则会阻塞主流程

### 3.5 代码知识图谱构建失败修复 (NEW)

**问题**: 日志显示"不支持的语言: javascript"，导致代码图谱构建异常

**根因**: tree-sitter-javascript和tree-sitter-typescript Python包未安装

**修复方案** (`zulong/tools/code_tools.py`):

```python
# 索引前检查语言包可用性
available_languages = []
for lang in languages:
    parser = ASTParser(lang)
    if parser.available:
        available_languages.append(lang)
    else:
        logger.warning(f"跳过语言 {lang}：tree-sitter 语言包未安装")

if not available_languages:
    logger.error("没有可用的语言包，索引终止")
    return

# 使用过滤后的语言列表
code_graph = bg_builder.build(
    resolved_root,
    languages=available_languages,  # 只索引可用的语言
    max_files=max_files,
)
```

**效果**:
- ✅ Python索引正常工作
- ✅ JS/TS被优雅跳过并记录警告
- ✅ 异常处理广播CRG_INDEX_FAILED事件

## 四、性能优化经验

### 4.1 任务图谱Web端性能优化 (NEW)

**问题**: Web页面加载超多节点(1763节点)任务图谱时严重卡顿

**优化方案** (Phase 1紧急修复):

#### 优化1: 边渲染视口裁剪 ⭐⭐⭐⭐⭐
```typescript
// 只渲染可见节点相关的边
const visibleEdges = edgeArray.filter(
    (e) => visibleSet.has(e.source) && visibleSet.has(e.target)
)
renderer.renderEdges(visibleEdges, positions)
```
**效果**: 减少50-80%边渲染开销

#### 优化2: 渐进式布局展示 ⭐⭐⭐⭐
```typescript
// 监听Web Worker的partial回调
client.onPartial((positions) => {
    setLayoutPositions(positions)
    setRenderStatus("rendering")  // 提前渲染
})
```
**效果**: 1-2秒内显示初步布局，避免长时间空白

#### 优化3: 批量更新防抖 ⭐⭐⭐
```typescript
// 500ms防抖，批量添加后只布局一次
layoutTimerRef.current = setTimeout(() => {
    performLayout()
}, 500)
```
**效果**: 减少70-90%重复布局计算

**效果**: 渲染速度提升30-50%

## 八、常见问题与解决方案

### 8.1 FC循环问题

**Q: 模型陷入死循环怎么办?**
A: Circuit Breaker会自动检测并熔断。检查日志中的`[CircuitBreaker]`信息。

**Q: 4B模型频繁忘记调用task_mark_status?**
A: RESUME AutoMark安全网会自动补标，最多5次。

**Q: 模型过早声明任务完成?**
A: RuleGuardian会拦截并注入纠正指令。

### 8.2 记忆系统问题

**Q: 记忆检索不准确?**
A: 检查MemoryGraph激活值同步，已完成任务应设为0.1。

**Q: 边权重不衰减?**
A: 确认`last_activated`字段存在，已修复fallback到`created_at`。

### 8.3 部署问题

**Q: Docker启动失败?**
A: 检查模型路径是否存在，日志目录权限是否正确。

**Q: GPU显存不足?**
A: 使用4bit量化模型，或切换到云端模型部署。

**Q: IDE连接失败?**
A: 确认8090端口未被占用，WebSocket服务已启动。

### 8.4 性能问题 (NEW)

**Q: 任务图谱加载大量节点时卡顿?**
A: 已实施Phase 1优化，2000节点从卡顿提升到30-40 FPS。如仍卡顿，检查浏览器性能和显存占用。

**Q: 节点审查任务无法打断?**
A: 已修复，L2BackupScheduler支持中断机制，用户发起新任务时会自动中断审查。

3. 编写单元测试
4. 理解系统级补偿机制

### 11.2 核心认知要点

1. **系统哲学**: "系统承担认知基础设施，模型只负责局部决策"
2. **架构优势**: "集中认知 + 分布式技能" 而非多Agent水平协作
3. **技术壁垒**: 5层FC防护链 + 图记忆 + 任务编排深度耦合
4. **部署灵活**: 支持本地/云端/混合/vLLM四种模式
5. **小模型补偿**: 8种机制让4B模型能做中等复杂度任务

### 11.3 常见误区

❌ **误区1**: 祖龙只是又一个Agent框架  
✅ **正解**: 祖龙是认知科学驱动的系统级AI，有独特的图记忆和FC防护链

❌ **误区2**: 祖龙只能用本地小模型  
✅ **正解**: 支持全场景部署，L2/L3都可用云端大模型

❌ **误区3**: 多Agent一定比单Agent强  
✅ **正解**: 取决于场景，祖龙的集中决策在端侧/实时场景更优

❌ **误区4**: 系统补偿能替代模型能力  
✅ **正解**: 系统能防止失控，但不能替代思考，超复杂任务仍需大模型
