## 二、问题根因分析

### 2.1 完整流程追踪

```
┌─────────────────────────────────────────────────────────────────┐
│ 阶段1: 任务空闲超时                                              │
└─────────────────────────────────────────────────────────────────┘
12:09:46  任务空闲900秒 → Gatekeeper检测到超时
              │
              ▼
         发布 SYSTEM_INTERRUPT 事件
              │
              ▼
         InferenceEngine._on_interrupt()
              │
              └─ self._interrupt_flag = True  ← ⚠️ 设置但未清除
              
┌─────────────────────────────────────────────────────────────────┐
│ 阶段2: 中断控制器处理                                            │
└─────────────────────────────────────────────────────────────────┘
12:09:46  InterruptController.on_interrupt()
              │
              ├─ 冻结当前任务（tg_1779065869）
              ├─ 创建新任务（task_a7b7976f...）
              └─ _process_interrupt() → 恢复tg_1779065869
              
         ⚠️ 关键问题：未清除 _interrupt_flag
              
┌─────────────────────────────────────────────────────────────────┐
│ 阶段3: 用户连接并恢复任务                                        │
└─────────────────────────────────────────────────────────────────┘
12:13:14  用户WebSocket连接（session=72ba0a910aac）
              │
              ▼
         session_resume() → 检测到活跃图tg_1779065869
              │
              ▼
         IDEFCRunner.run_or_resume_async() 启动
              │
              ├─ _check() 第一轮检查
              │   │
              │   └─ engine._interrupt_flag == True  ← ⚠️ 残留标志
              │       │
              │       └─ return "interrupted"
              │
              ▼
         FC循环立即终止，turns=1
              │
              ▼
         触发 CORE_TIMEOUT 降级处理
```

### 2.2 根因定位

**问题**: `InferenceEngine._interrupt_flag` 在中断时设置为`True`，但**从未被清除**。

**影响**:
1. 任务恢复后，FC循环的第一次`_check()`就检测到中断标志
2. 立即返回"interrupted"，FC循环无法运行
3. 由于没有实际调用模型（`elapsed_s=0.0`），触发降级处理

## 四、解决方案

### 4.1 推荐方案：在恢复任务时清除中断标志

**修改位置**: `zulong/ide/ide_server.py` 的 `session_resume()` 或 `IDEFCRunner` 初始化时

```python
# ide_fc_runner.py 中增加清除逻辑
def _init_state(self, messages):
    """初始化状态"""
    # 清除残留的中断标志
    if hasattr(self, 'engine') and self.engine:
        if getattr(self.engine, '_interrupt_flag', False):
            logger.info("[IDEFCRunner] 清除残留的中断标志")
            self.engine._interrupt_flag = False
    
    # 原有初始化逻辑
    state = IDEFCState(messages=messages)
    # ...
    return state
```

### 4.2 辅助方案：在中断处理完成时清除

**修改位置**: `zulong/l2/interrupt_controller.py:109-129`

```python
def _process_interrupt(self, event: ZulongEvent, task_id: str):
    """处理中断"""
    state_manager.set_l2_status(L2Status.BUSY, task_id)
    logger.info("Processing interrupt...")
    time.sleep(1)
    state_manager.set_l2_status(L2Status.IDLE)
    
    # ✅ 新增：清除中断标志
    if self._inference_engine:
        self._inference_engine._interrupt_flag = False
    
    # 检查任务栈并恢复
    task_stack = task_state_manager.get_task_stack()
    if task_stack:
        resumed_task_id = task_stack[-1]
        logger.info(f"Auto-resuming task {resumed_task_id} from stack")
        task_state_manager.resume_task(resumed_task_id)
```

### 4.3 完整方案：两处都清除

| 位置 | 时机 | 目的 |
|------|------|------|
| `interrupt_controller._process_interrupt()` | 中断处理完成时 | 确保中断处理完毕后标志清除 |
| `ide_fc_runner._init_state()` | FC循环初始化时 | 兜底清除残留标志 |

## 五、空闲超时分析

### 5.1 为什么会空闲超时？

**日志**: `[ZULONG] 任务 tg_1779065869 空闲超时 (900s >= 900s)`

**可能原因**:
1. **用户长时间无操作**: 15分钟（900秒）未发送消息
2. **WebSocket断开**: 前端连接中断，未及时重连
3. **任务阻塞**: FC循环卡在某一步骤（如API等待）

### 5.2 空闲超时配置

**文件**: `zulong/l1b/scheduler_gatekeeper.py`

```python
# 空闲超时阈值（可配置）
IDLE_TIMEOUT_SECONDS = 900  # 15分钟
```

**建议**:
- 对于长时间任务（如代码生成），可适当延长阈值
- 或者在任务执行期间定时更新活跃时间
